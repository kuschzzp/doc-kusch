# MCP 与 Skill 的区别：两种 Agent 能力扩展机制的设计对比

在构建 Agent 系统时，经常会遇到两类能力扩展机制：

- **Skill（技能）**：以 Claude Skills、OpenAI Skills、Codex Skills 等为代表。
- **MCP（Model Context Protocol）**：以协议化连接外部系统为核心。

它们都可以扩展大模型的能力边界，但解决的问题并不一样。

如果只用一句话概括：

> Skill 解决的是“怎么把一类任务的经验、流程和资源按需交给模型”；MCP 解决的是“怎么让模型通过标准协议访问外部系统能力”。

所以，Skill 和 MCP 不是替代关系，而是互补关系。

## 一、Skill 是什么？

### Skill 的本质

Skill 不是简单的一句 Prompt，也不只是一个函数。

更准确地说，Skill 是一个 **按需加载的任务能力包**。

它的入口通常是一个 `SKILL.md` 文件，用来告诉模型：

- 这个 Skill 适合什么任务。
- 遇到这类任务时应该遵循什么流程。
- 有哪些注意事项。
- 是否应该使用配套脚本、模板、参考资料或资源文件。

一个 Skill 目录里通常不只有 `SKILL.md`，还可以包含：

- 脚本。
- 模板。
- 示例。
- 参考文档。
- 静态资源。
- 配置文件。

所以不要把 Skill 简单理解成“提示词片段”。

它更像是：

> 一组为特定任务准备的说明书、工作流和配套资源。

### Skill 是如何被使用的？

Skill 的典型使用方式是 **Progressive Disclosure（渐进式披露）**。

也就是说，模型一开始通常不会看到所有 Skill 的完整内容，而是先看到一个简短的技能目录。

#### 阶段一：只暴露技能摘要

在对话或 Agent 启动时，模型看到的可能只是：

```text
You have the following skills:
- report-gen: Generate structured reports
- db-migrate: Help migrate database schemas
- code-review: Review code changes
```

此时模型只知道：

- 技能名称。
- 简要用途。
- 大概什么时候该使用。

它还不知道每个 Skill 的完整步骤和细节。

#### 阶段二：按需加载 Skill 内容

当模型判断当前任务需要某个 Skill 时，系统才会把对应的 `SKILL.md` 内容加载进上下文。

比如模型决定使用 `db-migrate`，系统才会提供类似内容：

```text
Steps:
1. Read current schema.
2. Identify breaking changes.
3. Generate migration plan.
4. Verify rollback strategy.
...
```

如果这个 Skill 还带有脚本或模板，模型也可以根据 `SKILL.md` 的说明去使用这些配套文件。

这就是 Skill 的核心价值：

> 不把所有知识一次性塞给模型，而是在需要时再展开细节。

### Skill 解决的是什么问题？

Skill 主要解决的是：

- Prompt 过长、重复。
- 任务流程难以复用。
- 团队经验难以沉淀。
- Agent 在复杂任务中缺少稳定步骤。
- 模型不知道某类任务应该遵循哪些约定。

比如：

- 写技术方案。
- 做代码审查。
- 生成 PPT。
- 分析表格。
- 修改 Word 文档。
- 处理某个团队固定格式的报告。

这些任务的难点往往不是“缺一个外部 API”，而是“模型需要知道怎么做这类任务”。

这时 Skill 很适合。

## 二、MCP 是什么？

MCP，全称 **Model Context Protocol**。

它不是 Prompt 技巧，而是一个连接协议。

更准确地说：

> MCP 是一套让 AI 应用以标准方式连接外部工具、资源和上下文的协议。

需要注意：MCP 不只包含 Tools。

官方 MCP 体系里常见能力包括：

- **Tools**：让模型执行某个动作，比如查询数据库、创建 issue、调用内部 API。
- **Resources**：让模型读取外部资源，比如文件、文档、数据库记录。
- **Prompts**：由服务端提供可复用提示模板。
- **Roots**：让客户端告诉服务端当前可访问的工作区边界。
- **Sampling**：允许服务端请求客户端侧模型完成生成。
- **Elicitation**：允许服务端向用户请求补充信息。

不过这篇文章主要讨论的是最常被拿来和 Skill 对比的部分：**MCP Tools**。

### MCP Tools 的本质

MCP Tools 更接近：

> 函数签名 + API 契约 + 协议化调用。

一个 MCP Tool 通常会描述：

- 工具名称。
- 工具描述。
- 输入参数 schema。
- 必填字段。
- 参数类型。
- 可选的结构化输出。
- 错误语义。

比如一个创建 GitHub Issue 的工具，可以被描述成：

```json
{
  "name": "create_issue",
  "description": "Create a GitHub issue",
  "inputSchema": {
    "type": "object",
    "properties": {
      "repo": {
        "type": "string",
        "description": "Repository name"
      },
      "title": {
        "type": "string",
        "description": "Issue title"
      },
      "body": {
        "type": "string",
        "description": "Issue body"
      }
    },
    "required": ["repo", "title"]
  }
}
```

模型在决定是否调用工具之前，需要知道这个工具叫什么、能做什么、需要哪些参数。

这和 Skill 的自然语言流程说明不一样。

MCP Tool 的目标是让模型能够以更稳定、更可校验的方式调用外部能力。

### MCP 解决的是什么问题？

MCP 主要解决的是：

- 工具接入方式不统一。
- 参数格式容易 hallucination。
- 工具调用结果难以标准化。
- 不同模型、不同客户端、不同服务之间难以复用工具。
- 外部系统能力无法用统一协议暴露给 AI 应用。

比如：

- 访问数据库。
- 查询 GitHub。
- 操作文件系统。
- 调用浏览器自动化。
- 读取企业知识库。
- 调内部业务系统。

这些任务的关键不是“模型应该怎么思考”，而是“模型如何可靠地访问一个外部系统”。

这时 MCP 更适合。

## 三、Skill 与 MCP 的核心差异

可以用一张表来概括：

| 维度 | Skill | MCP |
| --- | --- | --- |
| 本质 | 按需加载的任务能力包 | 连接模型与外部系统的协议 |
| 典型入口 | `SKILL.md` | MCP Server |
| 核心内容 | 指令、流程、脚本、模板、参考资料 | Tools、Resources、Prompts 等 |
| 主要目标 | 让模型知道“怎么做一类任务” | 让模型能标准化访问外部能力 |
| 约束方式 | 软约束，依赖模型遵循说明 | 协议约束，可校验输入参数 |
| 典型粒度 | 一类任务或工作流 | 一个外部能力或资源接口 |
| Token 策略 | 渐进式披露，按需展开 | 工具可调用前需要暴露工具定义 |
| 适合场景 | 写作、分析、代码审查、文档处理、复杂流程 | 数据库、GitHub、文件系统、浏览器、企业系统 |

更简单一点说：

- Skill 更关注 **任务方法论**。
- MCP 更关注 **外部能力连接**。

## 四、上下文注入方式不同

### Skill：按需展开

Skill 的优势在于，它不需要一开始就把所有细节塞进上下文。

模型先看到技能名称和描述，判断需要时再加载完整 Skill 内容。

这对 token 成本很友好，尤其是当你有很多技能，但每次任务只会用到其中一两个时。

比如一个 Agent 同时有这些 Skill：

- 写博客。
- 生成 PPT。
- 分析 Excel。
- 修改 Word。
- 代码审查。
- 数据库迁移。

如果每次都把所有 Skill 的完整说明塞进上下文，成本会很高，也会干扰模型判断。

Skill 的做法是：先给目录，需要哪个再展开哪个。

### MCP：工具可调用前需要暴露定义

MCP Tool 不太一样。

模型如果要稳定调用一个工具，至少需要提前知道：

- 工具名称。
- 工具用途。
- 输入参数结构。
- 必填字段。
- 参数类型。

否则模型就无法判断这个工具是否适合当前任务，也无法构造正确参数。

所以对于可调用的工具，工具定义通常需要在模型做 tool call 之前暴露给它。

这里要注意，不是说 MCP 的所有信息都必须永久塞进上下文。

实际系统里，MCP Server、工具列表、资源索引等可以由 Host 或 Client 管理，也可以根据当前任务只暴露相关工具。但只要某个工具要被模型调用，它的基本定义就必须对模型可见。

## 五、为什么 MCP 往往更消耗 Token？

MCP 并不是天然“浪费 token”，它的 token 成本来自它要表达更精确的能力边界。

一个工具定义通常需要包含：

- 名称。
- 描述。
- 参数 schema。
- 类型信息。
- 必填字段。
- 枚举值。
- 参数说明。

这些内容比一句自然语言说明更长。

比如 Skill 里可能只写：

```text
需要创建 GitHub issue 时，使用 GitHub 工具。
```

但 MCP Tool 需要告诉模型：

```text
工具名是什么，repo 怎么传，title 是否必填，body 是否可选，label 是字符串还是数组。
```

这就是 MCP 的 token 成本。

但这笔成本换来的是：

- 更高的 tool-call 成功率。
- 更低的参数 hallucination。
- 更容易做参数校验。
- 更容易跨客户端复用工具。
- 更容易接入真实外部系统。

所以它不是“更差”，而是在用 token 换确定性。

## 六、Skill 更省 Token，但代价是什么？

Skill 的低 token 成本主要来自 Progressive Disclosure。

它只在需要时加载完整内容，初始上下文通常只需要保留技能摘要。

但这也有代价：

- 触发依赖模型判断，可能该用时没用。
- Skill 内容是自然语言，约束力弱于 schema。
- 不适合表达严格参数接口。
- 很难像 MCP Tool 一样做静态参数校验。
- 如果 Skill 很大且频繁触发，实际 token 成本也不一定低。

所以不能简单说 Skill 一定更省 token。

更准确的说法是：

> Skill 更适合降低初始上下文成本；MCP 更适合提高外部调用的确定性。

## 七、Skill 和 MCP 怎么一起用？

成熟的 Agent 系统里，Skill 和 MCP 往往不是二选一，而是一起用。

举个例子：生成一份数据库迁移风险报告。

### Skill 负责工作流

Skill 可以定义：

- 先读取当前 schema。
- 再对比 migration 文件。
- 找出 breaking changes。
- 检查回滚策略。
- 评估数据兼容性。
- 最后按固定模板生成报告。

也就是说，Skill 告诉模型：

> 这类任务应该怎么做。

### MCP 负责外部能力

MCP 可以提供：

- 读取数据库 schema 的工具。
- 查询 Git diff 的工具。
- 读取 migration 文件的资源。
- 创建 GitHub Issue 的工具。
- 提交 PR 评论的工具。

也就是说，MCP 告诉模型：

> 你可以调用哪些外部系统能力。

组合起来就是：

```text
用户任务：生成数据库迁移风险报告
    -> Skill 提供分析流程、风险清单、报告模板
    -> MCP 提供数据库、Git、GitHub 等外部系统访问能力
    -> Agent 按 Skill 的流程，调用 MCP 工具完成任务
```

这才是更接近真实工程的用法。

## 八、什么时候用 Skill，什么时候用 MCP？

### 更适合 Skill 的场景

Skill 适合那些“流程重、经验重、格式重”的任务：

- 技术博客写作。
- 代码审查。
- 架构设计。
- 文档生成。
- PPT 制作。
- 表格分析。
- 需求拆解。
- 测试计划编写。

这些任务的关键是：模型需要知道一套稳定的方法和标准。

### 更适合 MCP 的场景

MCP 适合那些“外部系统重、参数结构重、确定性要求高”的任务：

- 查询数据库。
- 操作 GitHub。
- 访问文件系统。
- 调用浏览器。
- 查询企业知识库。
- 调用内部业务 API。
- 执行基础设施操作。

这些任务的关键是：模型需要可靠访问外部能力，并按结构化参数完成调用。

### 判断标准

可以用两个问题判断：

1. 这个需求主要是在教模型“怎么做一类任务”吗？如果是，优先考虑 Skill。
2. 这个需求主要是在让模型“访问某个外部系统”吗？如果是，优先考虑 MCP。

如果两个问题答案都是“是”，那就一起用。

## 九、常见误解

### 误解一：Skill 就是 Prompt

不准确。

Skill 的入口通常是自然语言说明，但它可以包含脚本、模板、参考文件和资源。它是一种任务能力包，不只是一段 Prompt。

### 误解二：MCP 就是工具 schema

不准确。

MCP Tools 会使用 schema 描述输入参数，但 MCP 本身是协议，覆盖 Tools、Resources、Prompts 等多种能力。

### 误解三：MCP 一定更耗 token

不绝对。

如果一个系统暴露了大量工具，而且每个工具 schema 都很复杂，MCP 的上下文成本确实会高。但客户端可以根据任务裁剪工具列表，降低暴露给模型的工具数量。

### 误解四：Skill 比 MCP 更弱

也不准确。

Skill 和 MCP 解决的问题不同。Skill 更适合沉淀流程和经验，MCP 更适合连接外部系统。拿 Skill 去做数据库 API 调用会别扭，拿 MCP 去表达复杂写作流程也不自然。

## 总结

Skill 和 MCP 都在扩展 Agent 的能力，但它们站的位置不同。

Skill 更像是：

> 给 Agent 的任务方法论和工作手册。

MCP 更像是：

> 给 Agent 连接外部世界的标准插座。

Skill 让模型更知道“怎么做”。

MCP 让模型更稳定地“调用外部能力”。

真正成熟的 Agent 系统，往往不是 Skill 或 MCP 二选一，而是：

> 用 Skill 组织任务流程，用 MCP 连接外部系统。
