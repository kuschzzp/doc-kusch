## MCP Prompts

最近在看 MCP（Model Context Protocol）时，我一开始对 `prompts` 这个能力很疑惑：

> 提示词还需要 MCP 吗？  
> 我直接在客户端里写 prompt 不就好了？

后来想明白了：**MCP Prompts 的价值不在于“提示词本身”，而在于让 MCP Server 可以标准化地暴露一组可复用、可发现、可传参的任务模板。**

### MCP Prompts 是什么？

简单说，MCP Prompts 是 MCP Server 提供给客户端的一组“公共 prompt 模板”。

比如一个 GitHub MCP Server 可以提供：

```text
/review_pr
/write_release_notes
/summarize_issues
```

一个数据库 MCP Server 可以提供：

```text
/explain_schema
/find_slow_queries
/generate_report_sql
```

客户端不需要提前知道这些 prompt 写了什么，只要通过 MCP 协议查询：

```python
prompts = mcp_client.list_prompts()
```

然后再按名字取用：

```python
messages = mcp_client.get_prompt(
    name="review_pr",
    arguments={
        "pr_number": 123
    }
)
```

最终客户端会把返回的 `messages` 发给大模型。

### 它在哪个阶段生效？

MCP Prompts 生效的阶段是：**客户端调用大模型之前**。

它不是模型运行中自动触发的东西，也不是像 tool 那样由模型决定调用。它更像是客户端在发起对话前，先从 MCP Server 拿一段标准化上下文。

用 Python 伪代码表示：

```python
# 1. MCP Server 定义一个 prompt
class GitHubMCPServer:
    def list_prompts(self):
        return [
            {
                "name": "review_pr",
                "description": "审查一个 Pull Request",
                "arguments": ["pr_number"]
            }
        ]

    def get_prompt(self, name, arguments):
        if name == "review_pr":
            pr_number = arguments["pr_number"]

            return [
                {
                    "role": "user",
                    "content": {
                        "type": "text",
                        "text": f"请审查 PR #{pr_number}，总结风险、改进点和测试建议。"
                    }
                }
            ]
```

客户端侧：

```python
# 2. 客户端发现 MCP Server 提供了哪些 prompts
prompts = mcp_client.list_prompts()

# 3. 用户选择 review_pr
prompt_messages = mcp_client.get_prompt(
    name="review_pr",
    arguments={"pr_number": 123}
)

# 4. 客户端把 prompt 生成的 messages 发给大模型
response = llm.chat(
    messages=prompt_messages
)
```

真正影响模型的是最后一步：

```python
response = llm.chat(messages=prompt_messages)
```

所以 MCP Prompts 的位置可以理解成：

```text
用户选择任务
   ↓
客户端向 MCP Server 获取 prompt
   ↓
MCP Server 返回标准化 messages
   ↓
客户端把 messages 发给大模型
   ↓
模型开始回答
```

### 它和 Tools、Resources 有什么区别？

MCP 里常见的三个能力可以这样理解：

```text
Tools     让模型“能做什么”
Resources 让模型“能看什么”
Prompts   让用户“可以一键启动什么任务”
```

对应到代码：

```python
# Tool：执行动作
result = mcp.call_tool("search_issues", {"keyword": "login error"})

# Resource：读取资料
doc = mcp.read_resource("repo://README.md")

# Prompt：生成发给模型的 messages
messages = mcp.get_prompt("review_pr", {"pr_number": 123})
```

Tool 更像“函数调用”，Resource 更像“上下文数据”，Prompt 更像“任务入口”。

### 为什么提示词也要 MCP？

如果只是一个个人项目，当然没必要。直接在代码里写 prompt 就很好。

MCP Prompts 真正有价值的地方在于：

1. **跟着 MCP Server 一起分发**

   一个 GitHub MCP Server 不只提供 `list_prs`、`get_diff` 这些工具，也可以顺便提供 `review_pr`、`write_release_notes` 这些推荐工作流。

2. **客户端可以自动发现**

   不同客户端都可以用 `prompts/list` 查询这个 Server 有哪些任务模板，而不用每个客户端自己硬编码。

3. **可以参数化**

   Prompt 不再是一段固定文本，而是可以接收参数：

   ```python
   mcp.get_prompt("review_pr", {
       "pr_number": 123,
       "focus": "security"
   })
   ```

4. **可以和 Tools、Resources 配合**

   一个 prompt 可以引导模型使用该 MCP Server 提供的工具和资源，形成更完整的工作流。

### 我的理解

MCP Prompts 可以粗略理解成“公共 prompt 管理机制”，但它不是一个中心化 prompt 平台。

更准确地说：

> MCP Prompts 是 MCP Server 对外暴露的一组公共任务模板，客户端可以发现、选择、传参并转换成发给大模型的 messages。

它解决的不是“怎么写提示词”，而是：

> 一组和某个 MCP Server 强相关的提示词，如何被不同客户端标准化发现和复用。

### 什么时候有用？

适合：

```text
发布通用 MCP Server
有一组固定领域工作流
希望客户端自动展示任务入口
希望 prompt 跟 tools/resources 一起分发
```

不太适合：

```text
只有一个客户端
只有一两个固定 prompt
prompt 和 MCP Server 的能力关系不强
不需要跨客户端复用
```

### 小结一下

MCP Prompts 不是 MCP 的核心执行能力，它更像 MCP Server 附带的“工作流菜单”。

小项目里它可能显得鸡肋；但当一个 MCP Server 面向多个客户端、多个用户、多个固定任务场景时，它就能让 prompt 从“散落在代码里的字符串”变成“可发现、可传参、可复用的协议能力”。
