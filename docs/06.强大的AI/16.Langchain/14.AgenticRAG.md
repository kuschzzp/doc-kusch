# Agentic RAG

普通 RAG 是固定流程：

```text
用户问题 -> 检索 -> 生成答案
```

Agentic RAG 是让 Agent 自己决定什么时候检索、检索什么、是否继续追问。

## 示例：把 Retriever 包成 Tool

```python
from langchain.agents import create_agent
from langchain_core.documents import Document
from langchain_core.tools import tool
from langchain_core.vectorstores import InMemoryVectorStore
from langchain_openai import OpenAIEmbeddings

embeddings = OpenAIEmbeddings(model="text-embedding-3-small")

docs = [
    Document(page_content="LangChain 的 create_agent 用于创建工具调用 Agent。"),
    Document(page_content="RAG 通过检索相关上下文增强模型回答。"),
    Document(page_content="LangSmith 可以追踪 Agent 和 RAG 的执行过程。"),
]

vectorstore = InMemoryVectorStore.from_documents(docs, embeddings)
retriever = vectorstore.as_retriever(search_kwargs={"k": 2})

@tool
def search_docs(query: str) -> str:
    """搜索 LangChain 相关文档。

    当用户询问 LangChain、RAG、Agent、LangSmith 的知识时使用。

    Args:
        query: 检索问题或关键词。
    """
    results = retriever.invoke(query)
    return "\n\n".join(doc.page_content for doc in results)

agent = create_agent(
    model="openai:gpt-4o-mini",
    tools=[search_docs],
    system_prompt="你是一个 LangChain 技术助手。回答前可以搜索文档。",
)

result = agent.invoke({
    "messages": [{"role": "user", "content": "create_agent 是做什么的？"}]
})

print(result["messages"][-1].content)
```

## Agentic RAG 适合什么？

适合：

- 问题复杂，需要多次检索
- Agent 需要判断是否检索
- 需要混合多个工具
- 需要边查边推理

不适合：

- 简单固定问答
- 只需要一次检索
- 对延迟要求极高

## 总结

普通 RAG 是固定链路。

Agentic RAG 是把检索能力交给 Agent，让它自主决定如何使用。
