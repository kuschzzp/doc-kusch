# Agent2Agent协议

> 官方地址：[A2A Protocol](https://a2a-protocol.org/latest/)  
> 本文基于 A2A latest 文档和 v1.0 规范整理。

先说结论：

- A2A 不是一个 Agent 框架。
- A2A 不是 MCP 的替代品。
- A2A 是一个让 Agent 与 Agent 之间可以标准化通信、发现能力、协作执行任务的开放协议。

如果只用一句话概括：

> MCP 解决的是 Agent 怎么调用工具，A2A 解决的是 Agent 怎么和另一个 Agent 协作。

## 一、A2A 到底是什么？

A2A，全称 **Agent2Agent Protocol**。

你可以把它理解成：

> 一个让不同 Agent 系统之间可以互相发现、互相调用、互相协作的通信协议。

它强调的是 **协议**，不是框架。

也就是说，A2A 并不关心你的 Agent 是用什么技术栈实现的：

- 你可以用 Python。
- 也可以用 Java、Go、Node.js。
- 你可以基于 LangGraph。
- 也可以基于 AutoGen、CrewAI、Semantic Kernel。
- 甚至你是公司内部自己写的一套 Agent 系统，也没关系。

只要你的 Agent 对外遵守 A2A 的协议规范，暴露标准化的能力说明和交互接口，其他 Agent 或客户端就可以用统一方式和你通信。

这件事的重要性在于：今天很多团队都在做 Agent，但很多 Agent 系统本质上还是“孤岛”。

- 接口不统一。
- 返回格式不统一。
- 能力声明不统一。
- 长任务处理方式不统一。
- 流式输出方式不统一。
- 文件、结构化数据、多轮任务的表达方式也不统一。

结果就是：

> 你做了很多 Agent，但它们彼此并不好协作。

A2A 想解决的，就是这个问题。

## 二、A2A 不是 MCP，它们解决的问题不一样

这是最容易混淆的地方。

很多人第一次看到 A2A，都会把它和 MCP 放在一起比较。但实际上，这两个东西不是一个层次的问题。

### MCP 是干什么的？

MCP 更偏向：

> Agent 如何调用工具、资源、外部系统。

比如：

- 调数据库。
- 读文件。
- 调浏览器。
- 执行 Python。
- 调内部 API。
- 访问知识库。

这些事情，本质上都是：

```text
Agent -> Tool / Resource
```

### A2A 是干什么的？

A2A 更偏向：

> Agent 如何把任务交给另一个 Agent。

比如：

- 主调度 Agent 发现 HR Agent 更适合处理请假问题。
- 财务 Agent 发现报销分析需要交给数据分析 Agent。
- 企业助理 Agent 把 IT 故障转给 IT Support Agent。
- 一个 SaaS Agent 调另一个第三方专业 Agent。

这些事情，本质上是：

```text
Agent -> Agent
```

### 最简单的一句话区分

你可以这么记：

- MCP：我怎么使用工具。
- A2A：我怎么使用另一个 Agent。

所以它们是互补关系，不是替代关系。

一种很自然的架构是：

```text
用户
  -> 主 Agent
      -> 通过 A2A 调用其他专业 Agent
          -> 每个专业 Agent 内部再通过 MCP 调用工具、数据库、浏览器、知识库
```

也就是说：

- Agent 内部通过 **MCP** 使用工具。
- Agent 外部通过 **A2A** 和其他 Agent 协作。

## 三、为什么 A2A 会重要？

因为未来真正有价值的，不只是“单个 Agent 很聪明”，而是：

> 多个专业 Agent 可以组成协同网络。

举个例子。

假设你做了一个企业办公助手，它本身能处理很多事情：

- 查年假余额。
- 查报销进度。
- 提 IT 工单。
- 搜知识库。
- 总结会议纪要。

再假设公司里还有别的 Agent：

- HR Agent。
- 财务 Agent。
- IT Support Agent。
- 数据分析 Agent。
- 会议纪要 Agent。

如果没有 A2A，你往往只能：

- 把所有能力硬塞进一个超级大 Agent。
- 或者手写很多内部 RPC / HTTP 接口。
- 每个系统都自己定义一套调用格式。
- 每接入一个新 Agent，都写一套新适配逻辑。

时间一长，系统会越来越乱。

而有了 A2A 之后，理论上可以变成：

1. 主 Agent 发现有哪些 Agent 可用。
2. 读取它们的 `AgentCard`，了解能力、入口、协议、安全要求。
3. 根据任务选择合适的 Agent。
4. 用标准方式发送 `Message` 或创建 `Task`。
5. 统一接收结果、状态更新、流式事件或异步通知。

这样，整个 Agent 生态才有“互联互通”的可能。

## 四、A2A 的核心思想：把 Agent 当成独立服务

这是理解 A2A 的关键。

A2A 并不是把另一个 Agent 当成一个“函数”或者一个“工具调用”。

它更像是把对方当成一个：

> 独立运行、内部实现可不透明、但能通过标准协议协作的智能体服务。

这点非常重要。

因为现实里的 Agent 系统并不一定是你自己写的：

- 可能是其他团队维护的。
- 可能是别的语言实现的。
- 可能是第三方平台提供的。
- 甚至可能是一个黑盒 SaaS Agent。

A2A 不要求你知道对方内部怎么工作。

你只需要知道：

- 它是谁。
- 它会什么。
- 它支持哪些交互方式。
- 它怎么接收任务。
- 它怎么返回结果。
- 它需要什么认证方式。

这就够了。

这也是为什么我认为 A2A 更像是 **Agent 时代的服务协作协议**。

## 五、A2A 里的几个核心概念

如果刚开始接触 A2A，最需要搞明白的是这几个对象：

- `AgentCard`
- `AgentSkill`
- `Message`
- `Part`
- `Artifact`
- `Task`
- `contextId`

### 1. AgentCard

`AgentCard` 可以理解成这个 Agent 的“名片”。

它是 Agent 对外暴露的一份描述信息，通常会包含：

- Agent 名称。
- Agent 描述。
- 服务地址。
- 版本。
- 支持的协议接口。
- 支持的输入输出模式。
- 支持的能力。
- 支持的技能列表。
- 是否支持流式。
- 是否支持 push notification。
- 是否有认证要求。
- 是否声明扩展能力。

别的 Agent 或客户端，在调用你之前，通常会先读取你的 `AgentCard`，判断你到底能干什么、怎么调用、是否需要认证。

所以你可以把 `AgentCard` 理解成：

> 整个 Agent 的对外说明书。

### 2. AgentSkill

`AgentSkill` 是很多人一开始最容易误解的地方。

很多人会问：

> 如果我内部有十几个 skill，是不是都得在 A2A 里一个个定义？

答案是：不一定。

A2A 里的 `AgentSkill`，不是你内部每个函数、每个工具、每个子流程的逐一映射。

它更像是：

> 你对外公开的能力目录。

比如一个办公助手内部可能有十几个具体能力：

- 查年假。
- 提请假。
- 查报销。
- 提报销。
- 创建 IT 工单。
- 查询 IT 工单。
- 查知识库。
- 总结会议纪要。
- 开在职证明。
- 查组织架构。

但对外完全可以只定义 4 个 `AgentSkill`：

- `hr_services`
- `expense_services`
- `it_support`
- `knowledge_assistant`

这才是更合理的建模方式。

换句话说：

> 内部 skill 是实现细节，A2A 的 AgentSkill 是对外能力标签。

### 3. Message

`Message` 表示一次通信消息。

它可以来自用户，也可以来自 Agent。一个 `Message` 通常包含：

- `messageId`：消息 ID。
- `role`：消息角色，比如 `user` 或 `agent`。
- `parts`：消息内容，由一个或多个 `Part` 组成。
- `contextId`：上下文 ID，用来关联同一个会话或任务上下文。
- `taskId`：如果消息属于某个任务，可以关联到对应任务。

如果一次请求可以很快完成，A2A 交互会很像一次普通消息请求：发出去，处理完，回一个结果。

比如：

- “我还有几天年假？”
- “VPN 连不上，帮我提个工单。”
- “帮我总结下这段文字。”

### 4. Part

`Part` 是 A2A 中承载具体内容的最小单元。

一个 `Message` 或 `Artifact` 不是只能包含文本，它可以由多个 `Part` 组成。

常见的 Part 类型包括：

- 文本内容。
- 文件引用。
- inline 文件数据。
- 结构化 JSON 数据。

这让 A2A 不局限于纯文本对话，而是可以支持文件、表单、结构化参数、多模态结果等场景。

举个例子，用户不只是说一句“分析这个文件”，还可以把一个 CSV 文件作为 `Part` 发给数据分析 Agent；Agent 返回时，也不只是回一句话，而是可以返回一份报告文件和一段结构化统计结果。

### 5. Artifact

`Artifact` 是 Agent 在处理任务过程中产生的可交付结果。

比如：

- 一份分析报告。
- 一个生成的图片。
- 一个 CSV 文件。
- 一段结构化 JSON。
- 一个可下载文档。

`Message` 更像交流过程中的一句话，`Artifact` 更像任务真正产出的交付物。

比如：

- 用户发出 `Message`：帮我分析这个月的报销数据。
- 数据分析 Agent 返回 `Artifact`：报销分析报告、异常报销清单、汇总图表。

### 6. Task

`Task` 表示一个可跟踪的工作单。

如果一件事不会马上完成，或者需要多轮协作、持续更新状态，就更适合用 `Task`。

比如：

- 生成一份 30 天报销分析报告。
- 整理大量会议纪要。
- 创建审批流并等待后续结果。
- 发起 IT 任务并异步跟踪状态。

`Task` 的关键点不只是“长任务”，而是它有状态、有上下文、可以持续更新。

一个 Task 可能经历类似这样的状态：

```text
submitted -> working -> input-required -> completed
```

也可能进入失败、取消、拒绝等状态。

所以你也可以把它理解为：

- `Message`：一次通信消息。
- `Task`：可持续跟踪的工作单。
- `Artifact`：任务过程中或最终产生的交付物。

### 7. contextId

`contextId` 用来标识一段上下文。

它解决的是一个很现实的问题：Agent 与 Agent 之间的协作不一定只有一问一答，很多时候是连续多轮的。

比如：

1. 主 Agent 问财务 Agent：“帮我分析这份报销数据。”
2. 财务 Agent 说：“缺少部门维度，请补充。”
3. 主 Agent 补充部门数据。
4. 财务 Agent 继续完成分析。

这些消息应该属于同一个上下文。`contextId` 就是用来把这些交互串起来的。

## 六、A2A 的交互模式

A2A 不只是简单的 request / response。

按照官方规范，它支持几类常见交互模式。

### 1. 普通消息请求

适合很快完成的任务。

比如：

> “我还有几天年假？”

Client 发送一个 `message/send`，Server 很快返回一个 `Message` 或 `Task`。

### 2. 流式任务执行

适合需要实时观察进度的任务。

比如：

> “帮我生成一份完整的市场分析报告。”

这个任务可能要经历搜索、整理、分析、生成、校验多个阶段。调用方不应该一直干等，而是可以订阅流式事件，持续接收状态更新或产物更新。

常见事件包括：

- `TaskStatusUpdateEvent`：任务状态变化。
- `TaskArtifactUpdateEvent`：任务产物更新。

### 3. Push Notification

流式连接适合短时间持续输出，但如果任务很久才完成，就不一定适合一直保持连接。

这时可以使用 Push Notification。

比如：

- 一个报告生成任务需要 30 分钟。
- 一个审批任务需要等待人工处理。
- 一个 IT 工单需要后续状态回调。

服务端可以在任务状态变化时，主动通知调用方配置好的地址。

### 4. 多轮任务协作

A2A 支持任务过程中继续补充输入。

比如远端 Agent 处理任务时发现信息不足，可以把任务状态变成 `input-required`，调用方再基于同一个 `contextId` 或 `taskId` 继续发送消息。

这点很重要，因为 Agent 协作不是传统 RPC。传统 RPC 往往是一次请求、一次响应；Agent 协作经常是多轮确认、多轮补充、多轮交付。

## 七、A2A 的数据交互到底长什么样？

从开发者视角看，A2A 交互最终会落到一组标准化的数据模型和协议操作上。

常见实现可以使用：

- JSON-RPC。
- gRPC。
- HTTP+JSON / REST。
- 自定义 Protocol Binding。

所以，不能简单说 A2A “本质上就是 JSON-RPC”。

更准确的说法是：

> A2A 定义了一组标准对象和操作，不同协议绑定负责把这些对象和操作映射到具体传输方式上。

为了便于理解，下面用 JSON-RPC 风格举一个例子。

假设另一个 Agent 想请求你的办公助手：

> “VPN 连不上，帮我创建一个 IT 工单。”

请求体大概会像这样：

```json
{
  "jsonrpc": "2.0",
  "id": "req-001",
  "method": "message/send",
  "params": {
    "message": {
      "messageId": "msg-001",
      "role": "user",
      "parts": [
        {
          "kind": "text",
          "text": "VPN 连不上，帮我创建一个 IT 工单"
        }
      ]
    }
  }
}
```

如果你的 Agent 很快处理完成，返回可能就是：

```json
{
  "jsonrpc": "2.0",
  "id": "req-001",
  "result": {
    "message": {
      "messageId": "msg-002",
      "role": "agent",
      "parts": [
        {
          "kind": "text",
          "text": "已为你创建 IT 工单：IT-2026-0008。"
        }
      ]
    }
  }
}
```

如果任务不能马上完成，服务端也可以返回一个 `Task`，后续再通过查询、订阅或通知机制持续更新状态。

## 八、Agent 是怎么被发现的？

A2A 的发现机制核心是 `AgentCard`。

常见方式有三种：

### 1. well-known 地址

服务可以在标准路径暴露 Agent Card，例如：

```text
/.well-known/agent-card.json
```

调用方可以先读取这个地址，获得 Agent 的能力说明、服务地址、协议接口、安全要求等信息。

### 2. Registry / Catalog

在企业内部，更常见的方式可能是维护一个 Agent Registry 或 Agent Catalog。

主 Agent 不需要挨个猜地址，而是从注册中心里查询：

- 公司里有哪些 Agent。
- 每个 Agent 负责什么领域。
- 当前版本是什么。
- 是否可用。
- 谁负责维护。
- 需要什么权限。

### 3. 直接配置

对于简单系统，也可以直接在配置里写死某个 Agent 的地址和 Agent Card。

这种方式最简单，但可扩展性较弱，更适合 demo、内部小系统或固定依赖关系。

## 九、如果我已经有一个在线运行的 Agent，怎么接入 A2A？

这是非常现实的问题。

很多人会担心：

> 难道我要把整个 Agent 重写一遍吗？

其实完全没必要。

最快的方式通常不是重写，而是：

> 在你原有 Agent 前面，加一层很薄的 A2A Adapter。

也就是：

- 原来的办公 Agent 继续照常运行。
- 新增一个 A2A 适配层。
- 对外说 A2A 协议。
- 对内继续调你原来的 Agent API。

架构上就会变成这样：

```text
A2A Client / 其他 Agent
    -> A2A Adapter
        -> 你现有的办公 Agent
            -> 内部工具 / MCP / 数据库 / 工作流
```

这层适配器通常要做几件事：

1. 定义 `AgentCard`。
2. 设计几个高层 `AgentSkill`。
3. 把 A2A 的 `Message` 转成你内部 Agent 能理解的请求。
4. 把内部 Agent 的返回结果转成 A2A 的 `Message`、`Task` 或 `Artifact`。
5. 把内部长任务状态映射到 A2A 的 Task 状态。
6. 处理认证、鉴权和错误码。
7. 根据需要支持 streaming 或 push notification。

这就是最现实、改造成本最低的方案。

## 十、接入 A2A 时最容易踩的坑

### 1. AgentSkill 粒度太细

不要把内部每一个函数、工具、子流程都暴露成一个 `AgentSkill`。

A2A 面向的是 Agent 协作，不是函数调用。如果 Skill 粒度太细，调用方反而很难理解你的 Agent 到底适合完成什么任务。

更推荐的做法是：

- 对外暴露业务能力。
- 对内保留实现细节。
- Skill 名称稳定，内部实现可以变化。

### 2. 把 A2A 当成普通 HTTP API

A2A 不是普通 CRUD API。

Agent 协作天然会遇到：

- 长任务。
- 多轮确认。
- 中途缺少输入。
- 产物逐步生成。
- 任务失败和重试。
- 权限不足需要授权。

如果只按普通 request / response 思路设计，很快就会卡住。

### 3. 忽略 Artifact

很多 Agent 系统一开始只返回文本，但真正进入业务场景后，经常需要返回文件、报告、表格、图片、结构化 JSON。

这时就应该使用 `Artifact`，而不是把所有东西都塞进一段 text 里。

### 4. 忽略安全边界

Agent 与 Agent 互调之后，权限问题会变得更复杂。

你需要认真考虑：

- 调用方是谁？
- 它有没有权限调用这个 Agent？
- 它有没有权限访问某类数据？
- 某个任务是否需要用户授权？
- 返回结果里是否包含敏感信息？

A2A 规范里包含安全方案、认证信息、扩展 Agent Card 等机制，但具体业务权限仍然需要你自己设计清楚。

### 5. 没有设计任务状态映射

如果你原有 Agent 已经有自己的任务系统，需要提前设计状态映射。

比如：

```text
内部状态 queued      -> A2A submitted
内部状态 running     -> A2A working
内部状态 waitingUser -> A2A input-required
内部状态 success     -> A2A completed
内部状态 failed      -> A2A failed
内部状态 cancelled   -> A2A canceled
```

这一步很容易被忽略，但它直接决定调用方能不能稳定跟踪任务。

## 十一、什么时候适合用 A2A？

适合用 A2A 的场景：

- 你有多个独立 Agent，需要互相协作。
- 你希望第三方 Agent 能调用你的 Agent。
- 你希望内部不同团队的 Agent 能统一接入。
- 你的 Agent 任务有长流程、多轮交互、异步结果。
- 你希望把 Agent 当成独立服务管理，而不是塞进一个大系统。

不一定需要 A2A 的场景：

- 你只有一个简单 Agent。
- 你的 Agent 只是在内部调用几个工具。
- 你的问题用 MCP 或普通函数调用就能解决。
- 你没有跨团队、跨系统、跨语言的协作需求。

简单说：

> 如果你的问题是“Agent 怎么调用工具”，优先看 MCP。  
> 如果你的问题是“Agent 怎么调用另一个 Agent”，再考虑 A2A。

## 总结

A2A 的价值，不在于它又定义了一套请求格式，而在于它把 Agent 之间协作时最容易混乱的东西标准化了：

- Agent 怎么声明自己是谁。
- Agent 怎么声明自己会什么。
- Agent 怎么被发现。
- Agent 怎么接收消息。
- Agent 怎么处理长任务。
- Agent 怎么返回中间状态和最终产物。
- Agent 怎么处理流式输出和异步通知。
- Agent 怎么表达认证、安全和扩展能力。

MCP 让 Agent 更容易使用工具。

A2A 让 Agent 更容易使用另一个 Agent。

如果说 MCP 是 Agent 连接外部能力的插座，那么 A2A 更像是 Agent 之间互相协作的通用语言。
