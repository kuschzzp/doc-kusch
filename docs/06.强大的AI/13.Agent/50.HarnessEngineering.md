## 演变过程

harness 这个单词的翻译是**控制并利用；（把动物）拴在一起（或拴到某物上）；给（马）套上挽具；连接，串联**，不难理解，它的含义就是以一种驾驭的
方式去控制与利用某些东西，那么用在AI领域，它就代表控制与利用AI。

先看一下AI工程的发展过程：
> `Prompt Engineering`(提示词工程)   
> `Context Engineering`(上下文工程)    
> `Harness Engineering`(精细的驾驭AI工程)

| 阶段 | 关注点 | 典型问题 | 解决方式 |
| --- | --- | --- | --- |
| Prompt Engineering | 如何表达任务 | 问得不清楚，回答不准确 | 优化提示词 |
| Context Engineering | 给模型什么信息 | 信息不足，模型不知道背景 | 检索、记忆、上下文组装 |
| Harness Engineering | 如何设计 Agent 的工作环境 | 会写但不稳定、不可控、不可验证 | 工具编排、仓库知识、机械化约束、验证机制、反馈循环 |

这三个阶段不是互相替代的关系，而是逐步叠加。

Prompt Engineering 解决的是表达问题。

Context Engineering 解决的是补充信息。

Harness Engineering 解决的是让 Agent 在复杂工程任务中稳定执行。

OpenAI 在 2026 年 2 月发布了一篇文章：[Harness engineering: leveraging Codex in an agent-first world](https://openai.com/index/harness-engineering/)。这篇文章不是在讨论一个新的提示词技巧，而是在讨论当 Codex 这类 Agent 真正参与软件工程时，工程师的工作方式会发生什么变化。

一句话概括就是：

> Humans steer. Agents execute.

人类负责定义目标、边界和判断标准，Agent 负责在被设计好的环境中执行任务。

## 为什么要开始关注 Harness Engineering ？

### Agent执行时常见的问题

- Agent 知道目标，但不知道项目真实的工程边界。
- Agent 可以写代码，但不一定知道如何启动项目、验证页面、读取日志。
- Agent 可以根据上下文生成答案，但上下文太大时又容易被无关信息干扰。
- Agent 可以修复一个局部问题，但如果没有架构约束，容易引入新的结构混乱。
- Agent 可以跑测试，但如果测试、日志、监控、CI 没有暴露给它，它就无法形成稳定的反馈闭环。
- Agent 可以重复已有模式，但如果仓库中已经存在坏模式，它也会把坏模式继续放大。

这些问题的共同点是：它们不是单纯靠 Prompt Engineering 就能解决的。

Prompt 可以告诉 AI 应该怎么做，但不能稳定保证 AI 每次都在正确的环境里做。Context 可以让 AI 知道更多信息，但信息越多，也越容易引入噪声。到了 Agent 阶段，AI 不再只是回答问题，而是会读文件、改代码、执行命令、打开浏览器、跑测试、提交 PR，这时真正重要的就变成了：我们如何设计一个可执行、可观察、可验证、可纠偏的工程环境。

这就是 Harness Engineering 需要解决的问题。

## OpenAI 文章里的核心实践

OpenAI 这篇文章讲的是一个很极端的实验：他们用 Codex 构建了一个内部产品，并且约束自己不手写代码。应用逻辑、测试、CI、文档、可观测性、内部工具都由 Codex 生成。人类不再直接贡献代码，而是把主要精力放在设计环境、表达意图、构建反馈循环上。

这个实验说明了一件事：当 Agent 的能力足够强以后，工程瓶颈不一定是“模型会不会写代码”，而是“环境有没有被设计到足够清晰，让 Agent 能稳定工作”。

### 1. 工程师角色发生变化

在传统软件开发中，工程师主要工作是写代码、调试代码、评审代码。

但在 Agent-first 的工作方式里，工程师更像是在搭建一套工作系统：

1. 把高层目标拆成更小的设计、实现、测试、评审任务。
2. 为 Agent 提供完成任务所需的工具。
3. 把隐含知识沉淀到仓库里。
4. 把架构规范变成可执行的约束。
5. 把失败原因转化成新的工具、文档或自动化检查。

OpenAI 文章里有一个很关键的判断：早期进展慢，并不是因为 Codex 不行，而是因为环境没有被充分定义。Agent 缺少工具、抽象和内部结构，就很难朝着高层目标持续推进。

所以 Harness Engineering 不是让 Agent “再努力一点”，而是当 Agent 失败时，反过来问：

> 它缺少什么能力？这个能力如何变得可见、可调用、可验证？

### 2. 让应用对 Agent 可读

如果 Agent 只能读代码，它对系统的理解是不完整的。

真实的软件不是一堆静态文件，它还包括运行状态、页面行为、接口请求、日志、指标、链路追踪、错误堆栈、性能数据等。如果这些信息只存在于人的浏览器、终端或监控平台里，对 Agent 来说就等于不存在。

OpenAI 的做法是让 Codex 可以直接观察运行中的应用：

- 每个 git worktree 都可以启动一个独立的应用实例。
- Agent 可以通过 Chrome DevTools Protocol 操作页面。
- Agent 可以读取 DOM 快照、截图和导航状态。
- Agent 可以观察运行时事件。
- Agent 可以查询日志、指标和 traces。

这样一来，任务就不再只是：

> 修改这个按钮的样式。

而是可以变成：

> 复现这个 UI 问题，观察页面状态，修改代码，重启应用，再验证问题是否消失。

这就是 Harness 的价值：它让 Agent 不只是生成代码，还能观察系统、验证行为、继续修正。

### 3. 仓库知识是事实来源

OpenAI 文章里有一个很实用的观点：

> 给 Codex 一张地图，而不是一本 1000 页的说明书。

他们尝试过把所有规则都写进一个巨大的 `AGENTS.md`，结果发现这种方式很容易失败：

- 上下文是稀缺资源，巨大的说明文件会挤占真正任务和代码的空间。
- 当所有规则都被写成“重要规则”时，Agent 反而不知道哪个更重要。
- 单体说明文档很容易过期。
- 一大块文本很难被机械化校验。

所以更合理的方式是：`AGENTS.md` 不做百科全书，而是做目录。

真正的知识应该沉淀在结构化的仓库文档里，例如：

- 架构说明
- 产品规格
- 设计文档
- 执行计划
- 质量标准
- 安全要求
- 可靠性要求
- 已知技术债

这样做的好处是，Agent 不需要一次性读完所有内容，而是可以从一个短小稳定的入口开始，根据任务逐步找到相关信息。

这也是 Context Engineering 和 Harness Engineering 的一个区别：

- Context Engineering 更关注“把什么信息放进上下文”。
- Harness Engineering 更关注“这些信息如何被组织、索引、维护和校验，使 Agent 能长期可靠地使用”。

### 4. 让架构约束可以机械化执行

只靠文档不能保证 Agent 写出的代码长期一致。

文档可以告诉 Agent “请保持架构清晰”，但如果没有自动检查，它仍然可能引入错误依赖、错误命名、过大的文件、缺失的边界校验，或者绕过已有抽象。

OpenAI 的做法是把架构规则变成机械化约束：

- 用自定义 lint 限制依赖方向。
- 用结构测试验证模块边界。
- 强制数据在边界处被解析和校验。
- 静态检查结构化日志。
- 约束 schema、类型、文件命名。
- 限制文件大小。
- 把修复建议直接写进 lint 错误信息中。

这一点非常关键。

在人类优先的开发流程里，过多规则可能显得繁琐。但在 Agent 优先的开发流程里，规则一旦被编码进工具，就可以在每一次 Agent 执行中自动生效。

也就是说，Harness Engineering 不是靠反复提醒 Agent “不要破坏架构”，而是让破坏架构这件事在工具层面过不去。

### 5. 建立反馈闭环

Agent 要可靠工作，不能只靠一次性生成。

OpenAI 文章中描述的 Codex 工作流更像一个闭环：

1. 验证当前代码库状态。
2. 复现问题。
3. 记录失败现象。
4. 实现修复。
5. 驱动应用验证修复结果。
6. 打开 PR。
7. 响应人类或其他 Agent 的 review。
8. 检测并修复构建失败。
9. 在需要判断时升级给人类。

这个闭环里最重要的不是“Agent 第一次就写对”，而是“Agent 写错之后能不能知道错在哪里，并继续修”。

如果没有测试、浏览器、日志、CI、review、错误恢复机制，Agent 的执行就是一次性的。它只能生成一个看起来合理的结果，但无法证明这个结果真的可靠。

Harness Engineering 要做的，就是把这些反馈通道接入 Agent 的工作环境。

### 6. 处理 Agent 带来的熵增

Agent 会放大已有模式。

如果仓库里有清晰的架构、稳定的命名、可靠的工具，它会沿着这些好模式继续生成。如果仓库里已经存在混乱、重复、临时方案，它也可能把这些坏模式继续复制下去。

OpenAI 把这个问题称为类似“垃圾回收”的过程。他们会把一些核心原则编码到仓库中，并让后台 Codex 任务定期扫描偏差、更新质量评分、打开重构 PR。

这对我们普通项目也很有启发：

不要只在项目失控之后再大规模重构，而是应该把质量规则变成持续运行的小检查。技术债像高息贷款，越早处理成本越低。

## 一个更具体的例子：AI 编程助手

假设我们让 AI 修复一个前端项目中的 bug。

如果只停留在 Prompt Engineering，我们可能会这样说：

> 请帮我修复这个页面的报错。

如果加入 Context Engineering，我们会把报错信息、相关组件代码、接口返回值、项目结构一起提供给 AI。

但如果要做到 Harness Engineering，就不只是“给信息”这么简单，而是要定义完整的工作环境：

1. 修改前必须读取相关文件的最新内容。
2. 不能覆盖用户已有改动。
3. 可以启动本地开发服务。
4. 可以打开浏览器复现问题。
5. 可以读取控制台错误、网络请求和页面截图。
6. 可以运行 lint、typecheck、unit test。
7. 测试失败后可以根据错误继续修复。
8. 涉及危险命令或高风险操作时必须请求人工确认。
9. 最终必须说明修改内容、验证结果和残留风险。

这个过程就是一种 Harness。

它像是给 Agent 配了一套工程工作台：有任务说明、有工具、有规则、有观察窗口、有测试、有反馈、有边界。Agent 不是自由发挥，而是在这个工作台里完成任务。

## 我对 Harness Engineering 的理解

结合 OpenAI 这篇文章，我认为 Harness Engineering 的重点不是“控制 AI”，而是“设计 AI 可以稳定工作的系统”。

它至少包括六件事：

1. **Intent**：把人的目标表达成 Agent 可以执行的任务。
2. **Context**：把必要信息组织成可发现、可维护、可验证的知识。
3. **Tools**：让 Agent 能直接使用工程工具，而不是依赖人类复制粘贴。
4. **Guardrails**：把安全边界和架构边界变成可执行约束。
5. **Feedback**：让 Agent 能观察结果、发现错误、继续迭代。
6. **Maintenance**：持续清理坏模式，避免 Agent 把混乱放大。

如果说过去的 AI 使用方式是“人问，AI 答”，那么 Agent 时代更像是“人设定方向，AI 在工程系统里执行”。

所以真正的竞争力可能不只是“谁更会写提示词”，而是“谁更会设计 Agent 的工作环境”。

## 总结

Harness Engineering 不是一个玄学概念，也不是单纯的提示词技巧。它来自一个很现实的问题：当 Agent 开始真正参与软件工程时，怎样才能让它稳定、可控、可验证地工作？

OpenAI 的实践给出的答案是：

- 让人类从写代码转向定义目标和反馈循环。
- 让应用、日志、指标、文档、架构规则都对 Agent 可读。
- 把仓库知识作为事实来源，而不是依赖散落在聊天记录和人脑中的隐含知识。
- 把重要规则变成 lint、测试、CI 这类机械化约束。
- 通过持续反馈和垃圾回收，避免 Agent 生成速度带来的系统熵增。

Prompt Engineering 让 AI 更好地回答。

Context Engineering 让 AI 知道更多背景。

Harness Engineering 则让 AI 能够在真实工程环境中持续完成事情。
