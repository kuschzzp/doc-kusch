# Python 依赖管理工具

Python 的依赖管理一直有点乱。

刚开始写 Python 时，很多人都是这样：

```bash
pip install requests
pip freeze > requirements.txt
```

这套方式能用，也足够简单。但项目一大，问题就会慢慢冒出来：

- 哪些包是我主动装的，哪些包只是被带进来的？
- 为什么我这里能跑，同事那里装完就报错？
- 开发依赖和线上依赖要不要分开？
- `requirements.txt`、`setup.py`、`pyproject.toml`、`poetry.lock`、`uv.lock` 到底谁说了算？

这篇不打算把所有历史讲成百科，重点是梳理：现在做 Python 项目，依赖管理到底应该怎么选。

## 先给结论

如果是新项目，我现在会优先考虑：

```text
能用 uv 就用 uv。
团队已经稳定使用 Poetry，就继续 Poetry。
非常小的脚本或临时项目，用 pip + requirements.txt 也没问题。
```

不要为了“现代化”把所有项目都迁移一遍。依赖管理工具的目标不是炫技，而是让环境可复现、协作成本低、CI 跑得稳。

## pip + requirements.txt：简单，但别对它要求太高

`pip` 是 Python 最基础的包安装工具，`requirements.txt` 是最传统的依赖列表。

常见用法：

```bash
pip install requests flask
pip freeze > requirements.txt
pip install -r requirements.txt
```

`requirements.txt` 适合什么？

- 临时脚本。
- 学习项目。
- 很小的 demo。
- 部署平台明确要求 `requirements.txt`。

它的问题也很明显。

`pip freeze` 导出来的是当前环境里所有包，包括间接依赖。比如你只主动装了 `requests`，但文件里会出现 `urllib3`、`certifi`、`idna` 这些包。时间一长，你已经分不清哪些是项目真正需要的，哪些只是某个旧版本顺手带进来的。

所以我一般不会把 `pip freeze` 当成项目依赖的长期维护方式。

如果只是小项目，可以手写一个干净点的 `requirements.txt`：

```txt
requests>=2.32,<3
flask>=3,<4
```

而不是把整个虚拟环境冻住。

## pyproject.toml：现在应该认识它

`pyproject.toml` 是现在 Python 项目的核心配置文件。

官方 Python Packaging 文档里也把它作为现代项目元数据和构建配置的入口。它可以放项目名、版本、Python 版本要求、依赖、可选依赖，也可以放工具配置，比如 ruff、pytest、mypy。

一个很普通的例子：

```toml
[project]
name = "myproject"
version = "0.1.0"
requires-python = ">=3.11"
dependencies = [
    "requests>=2.32,<3",
    "flask>=3,<4",
]

[project.optional-dependencies]
dev = [
    "pytest>=8,<9",
    "ruff>=0.8",
]
```

用 pip 也能安装：

```bash
pip install -e .
pip install -e ".[dev]"
```

不过要注意：`pyproject.toml` 只是声明项目依赖范围，它不是锁定文件。

也就是说：

```toml
requests>=2.32,<3
```

表达的是“允许安装这个范围内的版本”，不是“所有人都必须安装同一个精确版本”。

如果你要团队环境完全一致，还需要锁定文件。

## 锁定文件到底解决什么问题？

依赖文件一般分两层：

```text
pyproject.toml  写的是依赖约束
lock 文件       写的是解析后的精确版本
```

比如你在 `pyproject.toml` 写：

```toml
requests>=2.32,<3
```

今天解析出来可能是：

```text
requests==2.32.3
urllib3==2.2.3
certifi==2024.8.30
```

过几个月再装，可能就变了。

锁定文件的意义就是把“这一次解析出来的完整结果”记录下来。团队成员、CI、线上环境都按这个结果安装，少很多“我这里没问题”的争论。

常见锁定文件：

```text
poetry.lock   Poetry 生成
uv.lock       uv 生成
```

## Poetry：稳定，完整，但现在不是唯一选择

Poetry 曾经是 Python 项目依赖管理里体验最完整的方案之一。

它解决了几个实际问题：

- 用 `pyproject.toml` 管理项目。
- 用 `poetry.lock` 锁定依赖。
- 自动管理虚拟环境。
- 支持依赖分组。
- 支持打包和发布。

常用命令：

```bash
poetry init
poetry add requests
poetry add --group dev pytest
poetry install
poetry run python main.py
```

Poetry 项目里通常会看到：

```text
pyproject.toml
poetry.lock
```

`pyproject.toml` 写你想要什么，`poetry.lock` 写最终解析出来什么。

如果团队已经在用 Poetry，而且 CI、发布流程都跑得好，我不建议为了追新立刻迁移。Poetry 的生态和稳定性仍然不错。

但如果是新项目，我会更倾向先看 uv。

## uv：现在很值得优先考虑

uv 是 Astral 做的新一代 Python 包和项目管理工具，用 Rust 写，速度很快。

官方定位比较明确：它想覆盖以前很多工具分散承担的事情，比如 pip、pip-tools、pipx、poetry、pyenv、virtualenv、twine 等。

实际用起来，它最吸引我的点有三个：

1. 安装和解析速度快。
2. 项目工作流简单。
3. `uv.lock` 是跨平台锁定文件，适合提交到版本库。

新项目可以这样开始：

```bash
uv init
uv add requests flask
uv add --dev pytest ruff
uv run python main.py
```

常用命令：

```bash
uv sync          # 根据 pyproject.toml / uv.lock 同步环境
uv lock          # 更新锁定文件
uv run pytest    # 在项目环境里运行命令
uv tree          # 查看依赖树
uv remove flask  # 移除依赖
```

如果你手上已经有 `requirements.txt`，也可以：

```bash
uv pip install -r requirements.txt
```

或者把依赖加进项目：

```bash
uv add -r requirements.txt
```

我的判断是：如果没有历史包袱，uv 现在可以作为新 Python 项目的默认选择。

## 那到底怎么选？

我会按项目类型来选。

### 临时脚本

用 pip 或 uv 都行。

```bash
python -m venv .venv
source .venv/bin/activate
pip install requests
```

或者：

```bash
uv run --with requests script.py
```

如果只是一次性脚本，没必要搞一堆配置文件。

### 个人小项目

我会直接用 uv。

```bash
uv init
uv add requests
uv run python main.py
```

提交：

```text
pyproject.toml
uv.lock
```

### 团队业务项目

优先 uv 或 Poetry。

如果是新项目：

```bash
uv init
uv add fastapi sqlalchemy
uv add --dev pytest ruff mypy
```

如果团队已有 Poetry 规范，就继续：

```bash
poetry add fastapi sqlalchemy
poetry add --group dev pytest ruff mypy
poetry install
```

重点不是工具名字，而是要做到：

- 依赖写在 `pyproject.toml`。
- 锁定文件提交到 Git。
- CI 使用锁定文件安装。
- 不要手改 lock 文件。

### Python 库项目

库项目和应用项目不太一样。

应用项目追求环境完全一致，所以 lock 文件很重要。

库项目通常要支持一段依赖范围，让使用者在自己的环境里解析依赖。因此库的 `pyproject.toml` 应该写合理的版本范围，不要把依赖钉得太死。

至于 lock 文件要不要提交，不同团队习惯不一样。我的做法是：

- 如果 lock 文件只是给维护者本地开发和 CI 用，可以提交。
- 但不要让用户安装你的库时依赖这个 lock 文件。
- 发布到 PyPI 时，真正起作用的是包元数据里的依赖约束。

## 几个容易混的点

### 直接依赖和间接依赖

```text
你的项目
  └── requests        直接依赖，你主动选择
       ├── urllib3   间接依赖，requests 需要
       ├── certifi
       └── idna
```

`pyproject.toml` 里通常只写直接依赖。

lock 文件里会记录直接依赖和间接依赖的精确版本。

### requirements.txt 和 lock 文件不是一回事

`requirements.txt` 可以被当成锁定结果，也可以被当成手写依赖列表。问题就在这里：它的语义不够明确。

如果你用的是现代工具，我更愿意让：

```text
pyproject.toml 表达项目需要什么
uv.lock / poetry.lock 表达具体装什么版本
```

### 不要同时维护太多来源

有些项目里同时有：

```text
requirements.txt
requirements-dev.txt
pyproject.toml
poetry.lock
uv.lock
```

这不一定是好事。

文件越多，越容易出现“到底哪个才是准的”。

如果项目已经选择 uv，就让 `pyproject.toml + uv.lock` 成为主来源。

如果项目选择 Poetry，就让 `pyproject.toml + poetry.lock` 成为主来源。

除非部署平台必须要 `requirements.txt`，否则不要额外维护一份。

## 我的推荐

现在重新开一个 Python 项目，我会这样做：

```bash
uv init
uv add fastapi pydantic
uv add --dev pytest ruff mypy
uv run pytest
```

然后提交：

```bash
git add pyproject.toml uv.lock
git commit -m "init python project"
```

如果是老项目，先别急着迁移。先看三个问题：

1. 现在 CI 是否稳定？
2. 团队是否已经习惯 Poetry 或 pip-tools？
3. 迁移能不能明显减少安装时间或依赖冲突？

如果答案不明确，就先别动。

依赖管理工具应该减少麻烦，而不是制造新的麻烦。
