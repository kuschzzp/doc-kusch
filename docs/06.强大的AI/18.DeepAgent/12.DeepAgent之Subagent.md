# DeepAgent 之 Subagent

原文地址：[Deep Agents - Subagents](https://docs.langchain.com/oss/python/deepagents/subagents)

Subagent 这个东西，第一次看很容易理解成“多加几个工具人”。

但我觉得更准确的理解是：

> Subagent 是主 Agent 临时委托出去的一段隔离执行过程。

它不是普通 Tool，也不是长期在线的另一个 Agent。主 Agent 把一个相对完整的任务交给它，它自己在隔离上下文里执行，最后把结果汇报回来。

## 为什么需要 Subagent？

Deep Agent 处理的通常不是“一问一答”的任务，而是长任务。

比如：

- 连续搜索很多资料。
- 读多个文件。
- 分析一批日志。
- 对比几种方案。
- 让代码审查和文档整理同时推进。

如果所有事情都让主 Agent 自己做，主上下文会很快变脏。

搜索过程、文件片段、工具返回、中间判断都会混在一起。最后主 Agent 可能还没真正开始做决策，上下文里已经塞满了过程性垃圾。

Subagent 的价值就在这里：

```text
主 Agent 负责判断和调度
Subagent 负责把一段脏活做完
主 Agent 只接收整理后的结果
```

这就是官方文档里说的 context quarantine，上下文隔离。

我更喜欢把它理解成：主 Agent 不是什么都亲自干，而是学会把一坨会污染上下文的工作外包出去。

## Subagent 和 Tool 的区别

Tool 更像一个函数。

```text
输入参数 -> 执行动作 -> 返回结果
```

Subagent 更像一次小型任务执行。

```text
接收任务说明 -> 自己规划 -> 调用自己的工具 -> 汇总结果 -> 返回报告
```

所以不要把 Subagent 当成 Tool 的替代品。

如果只是查一次天气、算一个数字、读一个文件，用 Tool 就够了。

如果这个任务本身要经历好几步，而且中间过程主 Agent 没必要全部知道，就可以考虑 Subagent。

## 什么时候该用 Subagent？

我会用一个很简单的判断：

> 这个子任务如果放在主上下文里，会不会明显污染主任务？

会，就考虑 Subagent。

比较适合的场景：

- 研究型任务：搜索、筛选、归纳、给结论。
- 代码型任务：读代码、定位问题、给修改建议。
- 文件型任务：批量读文件、抽取重点、生成摘要。
- 分析型任务：跑工具、看结果、整理判断。
- 专业型任务：法律、财务、运维、安全等角色分工。

不太适合的场景：

- 一步就能完成的小动作。
- 主 Agent 必须逐条掌握中间过程。
- 子任务描述不清楚，需要频繁来回问。
- 只是为了“看起来像多 Agent”而拆分。

Subagent 会带来额外复杂度。能不用就不用，用的时候要有明确收益。

## 同步 Subagent 和异步 Subagent

官方这篇文档讲的是同步 Subagent。

同步的意思是：主 Agent 把任务交出去之后，会等 Subagent 执行完，再拿到结果继续往下走。

这适合大多数“委托一段工作，拿一个总结回来”的场景。

如果任务需要长时间运行、并行工作流、中途干预、取消任务，那就要看官方单独的 Async subagents。不要把所有委托场景都塞进同步 Subagent。

## 一个最小示例

官方推荐的自定义 Subagent 配置，本质上就是一个字典。

```python
from deepagents import create_deep_agent
from langchain.tools import tool

@tool
def search_docs(query: str) -> str:
    """Search internal documents."""
    return f"Found notes about {query}"

research_agent = {
    "name": "researcher",
    "description": "Researches a topic using document search and returns a concise summary.",
    "system_prompt": "You are a careful researcher. Search first, compare evidence, and return only the final summary with key sources.",
    "tools": [search_docs],
}

agent = create_deep_agent(
    subagents=[research_agent],
    system_prompt="You are a coordinator. Delegate research-heavy tasks to the researcher subagent.",
)
```

主 Agent 后续可以通过内置的 `task` 工具把任务交给 `researcher`。

你不需要自己实现“启动子 Agent”的底层逻辑，Deep Agents 已经内置了 SubAgentMiddleware。

## 配置字段不要只记 name/tools

一个字典形式的 Subagent，最重要的是这些字段：

| 字段 | 作用 | 继承关系 |
| --- | --- | --- |
| `name` | 唯一名称，主 Agent 调用 `task` 时会用到 | 必填 |
| `description` | 给主 Agent 判断是否委托 | 必填 |
| `system_prompt` | 子 Agent 自己的指令 | 必填，不继承主 Agent |
| `tools` | 子 Agent 可用工具 | 默认继承主 Agent；一旦指定就是完全覆盖 |
| `model` | 子 Agent 使用的模型 | 默认继承主 Agent |
| `skills` | 子 Agent 可加载的 skills | 自定义 Subagent 不继承，需显式配置 |
| `interrupt_on` | 子 Agent 内部工具是否需要人工确认 | 可覆盖主 Agent，使用时需要 checkpointer |
| `response_format` | 结构化输出 schema | 可选 |
| `permissions` | 文件系统权限规则 | 可选，设置后替换父级权限 |

这几个继承关系很容易混。尤其是 `tools`：不写时继承主 Agent 的工具；写了之后不是追加，而是覆盖。

## description 比你想的更重要

`description` 不是写给人看的简介，而是写给主 Agent 做路由判断用的。

写得太虚，主 Agent 就不知道什么时候该调用它。

不好的写法：

```python
"description": "Helpful assistant"
```

好一点的写法：

```python
"description": "Use for research-heavy questions that require searching documents, comparing evidence, and summarizing findings."
```

如果你发现 Subagent 经常不被调用，先别急着怪模型。很多时候是 description 没把边界说清楚。

要写清楚三件事：

- 它处理什么任务。
- 它什么时候应该被使用。
- 它和其他 Subagent 的区别是什么。

## system_prompt 要限制输出

Subagent 的 system prompt 不只是角色设定，更重要的是限制它怎么交付结果。

比如研究型 Subagent，我通常会要求它：

```text
1. 先搜索或读取资料。
2. 不返回原始大段内容。
3. 只返回结论、依据和不确定点。
4. 如果证据不足，明确说证据不足。
```

原因很简单：如果 Subagent 把十几页搜索结果原样返回，主上下文还是会被撑爆。那就失去了上下文隔离的意义。

所以 Subagent 的输出应该像报告，不应该像日志。

## tools 要少而准

Subagent 的工具不是越多越好。

一个研究型 Subagent 给搜索工具就够了。

一个代码审查 Subagent 给读文件、grep、测试工具就够了。

不要为了方便给它塞一堆工具：

```python
# 不建议
tools=[web_search, read_file, write_file, send_email, deploy, execute_sql]
```

工具越多，决策空间越大，出错面也越大。

更好的做法是按职责切：

```python
research_agent = {
    "name": "researcher",
    "description": "Researches topics using search.",
    "tools": [web_search],
}

code_reviewer = {
    "name": "code-reviewer",
    "description": "Reviews code changes and reports risks without editing files.",
    "tools": [read_file, grep],
}
```

Subagent 应该是专用工种，不是另一个全能主 Agent。

## 结构化输出

如果 Subagent 的结果要被主 Agent 继续处理，不要只依赖自由文本。

官方现在支持给 Subagent 配 `response_format`。这样 Subagent 返回给父 Agent 的内容会是符合 schema 的 JSON，而不是一段不稳定的自然语言。

例如：

```python
from pydantic import BaseModel, Field

class ResearchFindings(BaseModel):
    summary: str = Field(description="Summary of findings")
    confidence: float = Field(description="Confidence score from 0 to 1")
    sources: list[str] = Field(description="Source URLs")

research_agent = {
    "name": "researcher",
    "description": "Researches topics and returns structured findings.",
    "system_prompt": "Research the topic and return structured findings.",
    "tools": [web_search],
    "response_format": ResearchFindings,
}
```

这个能力适合“父 Agent 后续还要根据结果做判断”的场景。

如果只是给用户看，普通文本也够。

## general-purpose Subagent

Deep Agents 默认有一个同步 `general-purpose` Subagent。

它使用和主 Agent 类似的配置，但有独立上下文。

什么时候有用？

当你只是想隔离一段复杂工作，但这个工作又没有必要专门定义一个角色时，可以用它。

比如：

```text
把这几个文件读一遍，总结出跟登录逻辑有关的风险点。
```

这类任务不一定需要单独建 `login-reviewer`，让 `general-purpose` 去隔离处理就行。

但如果某类任务会反复出现，还是应该定义一个明确的 Subagent。因为明确的 description、tools 和 system_prompt 会让行为更稳定。

官方还提供了两个相关能力：

- 如果你在 `subagents` 里定义了一个同名的 `general-purpose`，会替换默认版本。
- 如果确实不想要默认的 `task` 委托能力，可以通过 harness profile 里的 `GeneralPurposeSubagentProfile(enabled=False)` 关闭默认 general-purpose，并且不传同步 subagents。

一般项目里不用急着禁用它。只有当你明确不希望主 Agent 做任何委托时，才需要关。

## runtime context 会传给 Subagent

官方文档里还有一个细节：父 Agent 调用时传入的 runtime context，会自动传给 Subagent。

也就是说，如果你的工具依赖 `user_id`、`session_id`、权限信息，这些上下文在 Subagent 的工具里也能拿到。

如果不同 Subagent 需要不同配置，可以在 context 里用明确字段区分，比如：

```python
from dataclasses import dataclass

@dataclass
class Context:
    user_id: str
    researcher_max_depth: int | None = None
    fact_checker_strict_mode: bool | None = None
```

不要在工具里靠猜测当前是谁调用的。如果确实要区分，可以读运行 metadata 里的 `lc_agent_name`。官方在 streaming 和 tracing 里也用这个字段标识是哪一个 agent 发起的调用。

## 不要假设 Subagent 会记住上一次调用

这是一个很容易踩的点。

Subagent 的任务应该按“一次委托给完整说明”来设计。不要指望它像主会话一样自然记住上一次委托里做了什么。

不好的使用方式：

```text
第一次：researcher，帮我查一下 A。
第二次：researcher，刚才你查到了什么？
```

第二次不一定知道第一次发生过什么。

更稳的方式是一次性把任务说完整：

```text
researcher，查 A 的资料，整理关键结论、证据来源和不确定点，最后返回一份摘要。
```

如果确实需要跨步骤共享结果，就让 Subagent 把结果写入文件、store，或者由主 Agent 接收结果后继续传递。

## Skills 不会自动继承

如果主 Agent 配了 skills，不代表自定义 Subagent 自动拥有这些 skills。

需要的话要显式配置。

```python
agent = create_deep_agent(
    skills=["/main-skills/"],
    subagents=[
        {
            "name": "tester",
            "description": "Writes and reviews Python tests.",
            "system_prompt": "Use the testing skill when writing test plans.",
            "skills": ["/testing-skills/"],
        }
    ],
)
```

这一点和默认 `general-purpose` 不太一样，写自定义 Subagent 时要注意。

## Streaming 里怎么区分是谁在调用工具？

当主 Agent 和多个 Subagent 共用工具时，trace 很容易混。

官方在 streaming / tracing metadata 里提供了 `lc_agent_name`。你可以用它区分某次工具调用来自主 Agent，还是某个 Subagent。

这对排查问题很有用：

```text
main-agent 调用了哪些工具
research-agent 调用了哪些工具
code-reviewer 调用了哪些工具
```

如果你接了 LangSmith，这个字段也能帮助看清楚执行链路。

## CompiledSubAgent 什么时候用？

大多数时候，字典形式的 Subagent 已经够用。

`CompiledSubAgent` 适合更复杂的情况：子任务本身就是一个 LangGraph。

比如：

- 有固定多节点流程。
- 有条件分支。
- 有循环重试。
- 有人类审批。
- 有明确状态机。

这种情况下，Subagent 不再只是“一个带 prompt 和 tools 的临时 Agent”，而是一个已经编译好的 runnable graph。

普通场景先别上来就用这个。先用字典形式，把职责和输出跑顺。

## 我会怎么设计 Subagent？

我一般按这几个问题来写：

1. 这个 Subagent 只负责哪类任务？
2. 主 Agent 什么时候应该调用它？
3. 它需要哪些最小工具？
4. 它最后应该返回什么格式？
5. 它不应该返回什么？

比如研究型 Subagent：

```python
research_agent = {
    "name": "researcher",
    "description": "Use for research-heavy questions that require multiple searches, source comparison, and synthesis.",
    "system_prompt": "Search thoroughly. Return a concise report with: conclusion, key evidence, source notes, and uncertainty. Do not return raw search dumps.",
    "tools": [web_search],
}
```

比如代码审查 Subagent：

```python
code_review_agent = {
    "name": "code-reviewer",
    "description": "Use for reviewing code changes, identifying bugs, regressions, and missing tests. It should not edit files.",
    "system_prompt": "Review code like a senior engineer. Return findings ordered by severity with file references. Do not rewrite the whole solution unless asked.",
    "tools": [read_file, grep],
}
```

这比简单写“research helper”“code helper”要稳定得多。

## 总结

Subagent 的核心价值不是“多 Agent 很酷”，而是隔离复杂任务。

主 Agent 应该负责：

- 判断任务怎么拆。
- 决定委托给谁。
- 汇总结果。
- 做最终决策。

Subagent 应该负责：

- 在隔离上下文里完成一段专业任务。
- 使用自己需要的最小工具集。
- 返回干净、压缩、可用的结果。

如果一个 Subagent 的 description 很模糊、工具很多、输出又不受限制，它大概率只会制造新的混乱。

真正好用的 Subagent，边界一定很窄，交付一定很清楚。
