# 垃圾回收 GC

官方参考：

- Oracle JDK GC Tuning Guide：[Available Collectors](https://docs.oracle.com/en/java/javase/21/gctuning/available-collectors.html)
- Oracle JDK GC Tuning Guide：[Garbage-First Garbage Collector](https://docs.oracle.com/en/java/javase/21/gctuning/garbage-first-garbage-collector.html)
- Oracle JDK GC Tuning Guide：[Z Garbage Collector](https://docs.oracle.com/en/java/javase/21/gctuning/z-garbage-collector.html)
- OpenJDK Shenandoah Wiki：[Shenandoah](https://wiki.openjdk.org/display/shenandoah)

GC 这块很容易被讲成一堆名词：Serial、Parallel、CMS、G1、ZGC、Shenandoah。

我觉得更好的理解方式是：先知道 GC 要解决什么问题，再看它是怎么一代代演进的。

一句话：

> GC 的目标是在尽量少影响业务线程的前提下，自动回收不再使用的对象内存。

不同收集器的差异，主要就在两个目标之间取舍：

```text
吞吐量：业务线程运行时间占比更高
低延迟：单次停顿时间更短、更稳定
```

## 一、GC 到底回收什么？

Java 是托管内存语言。对象大多分配在堆里，程序员不需要手动 `free`，JVM 会自动判断哪些对象已经没人用了，然后回收这些内存。

GC 最核心的问题是：

```text
哪些对象还活着？
哪些对象已经可以回收？
```

Java 使用的是可达性分析。

从一组 GC Roots 出发，能一路引用到的对象就是存活对象；到不了的对象就是垃圾。

常见 GC Roots：

- 虚拟机栈里的局部变量引用。
- 方法区 / 元空间里的类静态字段引用。
- 常量引用。
- JNI 本地方法引用。
- JVM 内部持有的一些引用。

## 二、为什么要分代？

大部分 Java 对象有一个特点：朝生夕死。

很多对象创建出来只活很短时间，比如临时字符串、请求对象、集合中间结果。

所以早期 JVM 的主流设计是分代收集：

```text
新生代：新对象先放这里，回收频繁
老年代：活得久的对象晋升到这里，回收较少
```

Java 8 里常见的堆结构是：

```text
Young Generation
  ├── Eden
  ├── Survivor 0
  └── Survivor 1
Old Generation
Metaspace
```

对象通常先进入 Eden。

Minor GC 后还活着，就在 Survivor 区之间复制。

活过多次之后，晋升到老年代。

这就是很多 Java 8 GC 知识的基础。

## 三、Java 8 时代：Parallel、CMS、G1 并存

很多老项目还停在 Java 8，所以要先从 Java 8 讲。

### Serial GC

开启方式：

```bash
-XX:+UseSerialGC
```

特点：

- 单线程收集。
- GC 时暂停所有应用线程。
- 实现简单。
- 小堆、小应用、客户端场景还能用。

它不是“性能差”的代名词。小内存、小服务里，Serial GC 的简单反而可能是优势。

但对大多数服务端应用，它不是首选。

### Parallel GC

开启方式：

```bash
-XX:+UseParallelGC
```

Java 8 服务端模式下，Parallel GC 是很常见的默认选择。

特点：

- 多线程回收。
- 关注吞吐量。
- GC 停顿时还是 Stop-The-World。
- 适合后台任务、批处理、对延迟不敏感但追求总处理量的应用。

它的思路很直接：

```text
停就停，但尽快用多线程把垃圾收完
```

如果你的应用是离线计算、批处理、导入导出任务，Parallel GC 仍然可以考虑。

### CMS

开启方式：

```bash
-XX:+UseConcMarkSweepGC
```

CMS，全称 Concurrent Mark Sweep。

它的目标是降低老年代回收停顿。

特点：

- 并发标记。
- 并发清除。
- 停顿时间比传统老年代收集器短。
- 不做压缩，容易产生内存碎片。
- 可能出现 concurrent mode failure。

CMS 曾经是低延迟服务端应用的常见选择。

但它的问题也很明显：碎片、调参复杂、维护成本高。

后续 JDK 已经逐步淘汰 CMS，G1 成为更主流的服务端选择。

### G1 GC

开启方式：

```bash
-XX:+UseG1GC
```

G1 在 JDK 7 引入，JDK 8 已经可用，JDK 9 开始成为默认 GC。

G1 的核心变化是：它不再把堆简单看成连续的新生代和老年代，而是拆成很多 Region。

```text
Heap
  ├── Region
  ├── Region
  ├── Region
  └── Region
```

每个 Region 可以扮演 Eden、Survivor、Old、Humongous 等角色。

G1 的目标是：

```text
在可预测停顿时间目标下，优先回收收益最大的 Region
```

所以它叫 Garbage First。

常见配置：

```bash
-XX:+UseG1GC
-XX:MaxGCPauseMillis=200
```

注意：`MaxGCPauseMillis` 是目标，不是硬保证。

G1 适合大多数普通后端服务，尤其是堆比较大、希望停顿更稳定的应用。

## 四、Java 9：G1 成为默认 GC

JDK 9 一个重要变化：默认 GC 从 Parallel GC 切到 G1。

这说明 JVM 默认选择从“优先吞吐量”转向了“更平衡的服务端表现”。

G1 更适合现在常见的 Web 服务：

- 堆比以前更大。
- 请求延迟更重要。
- 服务需要长期运行。
- 希望 GC 停顿更可控。

这不是说 Parallel GC 不能用了，而是默认值更贴近服务端应用的普遍需求。

## 五、Java 11：Epsilon 和 ZGC 出现

JDK 11 是一个 LTS 版本，这个版本开始出现两个很有代表性的 GC：Epsilon 和 ZGC。

### Epsilon GC

开启方式：

```bash
-XX:+UnlockExperimentalVMOptions -XX:+UseEpsilonGC
```

Epsilon 很特殊：它不回收垃圾。

对象照样分配，但不会释放。堆满了就 OOM。

它有什么用？

- 性能基准测试，排除 GC 干扰。
- 极短生命周期任务，程序在堆满前就结束。
- JVM / GC 实验。

生产服务基本不会用它。

### ZGC

JDK 11 引入 ZGC，最开始是实验特性；JDK 15 成为生产可用。

ZGC 的目标很明确：低延迟。

官方文档里强调，ZGC 的停顿时间通常非常短，并且设计上希望停顿时间不随堆大小线性增长。

早期 ZGC 是非分代的。

特点：

- 大部分工作并发执行。
- 支持很大的堆。
- 停顿时间很短。
- 更适合延迟敏感应用。

开启方式：

```bash
-XX:+UseZGC
```

如果你的服务对 P99、P999 延迟非常敏感，ZGC 就值得关注。

## 六、Java 12 到 15：Shenandoah 和 ZGC 走向成熟

### Shenandoah GC

Shenandoah 的目标和 ZGC 类似：低停顿。

它最大的特点是并发整理 / 并发移动对象，尽量减少应用线程停顿。

开启方式：

```bash
-XX:+UseShenandoahGC
```

Shenandoah 在不同 JDK 发行版中的支持情况要注意。OpenJDK、Red Hat 系列里更常见，Oracle JDK 的支持历史和其他发行版不完全一样。

它适合：

- 低延迟服务。
- 大堆应用。
- 希望压缩堆但不想长时间 STW 的场景。

### ZGC 正式可用

JDK 15 中，ZGC 从实验特性变成正式特性。

这意味着低延迟 GC 开始不再只是实验选项。

如果从 Java 8 升级到 Java 17 / 21，ZGC 是很值得重新评估的选择。

## 七、Java 17：低延迟 GC 进入 LTS 视野

Java 17 是 LTS。

这个阶段，主流选择大概是：

```text
G1：默认、通用、稳妥
ZGC：低延迟、大堆
Shenandoah：低延迟，视发行版支持而定
Parallel：吞吐优先
Serial：小应用、小堆
```

对普通后端服务来说，G1 仍然是很稳的默认选择。

不要一上来就为了“先进”切 ZGC。先看指标：

- GC 停顿是否真的影响业务？
- P99 延迟是否被 GC 拉高？
- 堆大小是否足够大？
- CPU 是否有余量给并发 GC？

## 八、Java 21：Generational ZGC

JDK 21 又是一个 LTS，而且 ZGC 有一个关键升级：Generational ZGC，分代 ZGC。

早期 ZGC 不分代。

但 Java 对象确实有“多数对象很快死亡”的特点，所以分代仍然有价值。

Generational ZGC 把 ZGC 的低延迟目标和分代假设结合起来：

```text
年轻对象更频繁回收
老对象较少扫描
仍然保持低停顿目标
```

开启方式：

```bash
-XX:+UseZGC -XX:+ZGenerational
```

在更新的 JDK 中，Generational ZGC 已经成为 ZGC 的主要方向。具体默认行为要以你使用的 JDK 版本官方文档为准。

如果是 Java 21 及以后，并且你关注低延迟，优先看 Generational ZGC，而不是老的非分代 ZGC。

## 九、几个常见 GC 概念

### Minor GC / Young GC

主要回收年轻代。

对象大多朝生夕死，所以 Young GC 通常发生频繁，但单次成本相对低。

### Major GC / Old GC

主要处理老年代。

不同收集器叫法和行为不完全一样，不要只看名字，要结合 GC 日志判断。

### Full GC

通常表示一次比较重的全堆级别回收，往往伴随较长 STW。

如果线上频繁 Full GC，基本都要排查。

### Stop-The-World

GC 过程中暂停所有 Java 应用线程。

低延迟 GC 的目标不是“完全没有 STW”，而是把 STW 时间压得很短。

### 吞吐量和延迟

吞吐量优先：任务总完成量更高，可以接受偶发停顿。

低延迟优先：单次请求停顿要短，P99/P999 更重要。

## 十、常用参数

GC 参数不要一上来就背一堆。

先分成四类看：

```text
堆大小参数
选择垃圾收集器的参数
控制 GC 行为的参数
GC 日志参数
```

### 1. 堆大小：-Xms 和 -Xmx

```bash
-Xms2g
-Xmx2g
```

这两个是最常见的 JVM 参数。

| 参数 | 含义 |
| --- | --- |
| `-Xms` | JVM 启动时的初始堆大小 |
| `-Xmx` | JVM 允许使用的最大堆大小 |

比如：

```bash
-Xms2g -Xmx2g
```

意思是：JVM 一启动就申请 2G 堆，最大也只允许用 2G。

为什么很多服务端项目建议两个值设一样？

因为如果不一样，JVM 运行过程中可能会扩容或收缩堆。这个过程本身也有成本，还可能让 GC 行为不稳定。

所以线上服务常见写法是：

```bash
-Xms4g -Xmx4g
```

但也不是越大越好。

堆太小：GC 频繁，甚至 OOM。

堆太大：单次回收成本可能变高，容器内存也可能被打爆。

如果部署在容器里，还要注意容器内存限制。不要只看宿主机内存。

### 2. 新生代比例：-XX:NewRatio

```bash
-XX:NewRatio=2
```

这个参数控制新生代和老年代的比例。

`NewRatio=2` 表示：

```text
老年代 : 新生代 = 2 : 1
```

也就是说，整个堆大概三分之一是新生代，三分之二是老年代。

什么时候关注它？

如果你的应用创建大量短命对象，Young GC 很频繁，可能需要关注新生代大小。

不过现在如果使用 G1，通常不建议一开始就手动调太多代际比例。G1 会根据停顿目标动态调整很多东西。

### 3. Eden 和 Survivor 比例：-XX:SurvivorRatio

```bash
-XX:SurvivorRatio=8
```

新生代通常包含：

```text
Eden + Survivor0 + Survivor1
```

`SurvivorRatio=8` 表示：

```text
Eden : 单个 Survivor = 8 : 1
```

因为有两个 Survivor，所以大概是：

```text
Eden : S0 : S1 = 8 : 1 : 1
```

什么时候关注它？

如果对象从 Eden 回收后，Survivor 放不下，会提前晋升到老年代。老年代对象变多，后续 Old GC 压力就会上来。

但同样，现代 GC 尤其是 G1 下，不建议没数据就乱调。

### 4. 选择垃圾收集器

这几个参数是“选择使用哪个 GC”。

```bash
-XX:+UseSerialGC
-XX:+UseParallelGC
-XX:+UseG1GC
-XX:+UseZGC
-XX:+UseShenandoahGC
```

含义：

| 参数 | 选择的 GC | 常见场景 |
| --- | --- | --- |
| `-XX:+UseSerialGC` | Serial GC | 小内存、小工具、单核环境 |
| `-XX:+UseParallelGC` | Parallel GC | 批处理、吞吐量优先 |
| `-XX:+UseG1GC` | G1 GC | 大多数服务端应用 |
| `-XX:+UseZGC` | ZGC | 低延迟、大堆应用 |
| `-XX:+UseShenandoahGC` | Shenandoah GC | 低延迟应用，视 JDK 发行版支持 |

一般不要同时写多个 GC 选择参数。

比如不要这样：

```bash
-XX:+UseG1GC -XX:+UseZGC
```

你应该先明确自己要用哪个。

### 5. G1 停顿目标：-XX:MaxGCPauseMillis

```bash
-XX:MaxGCPauseMillis=200
```

这个参数表示：希望 GC 停顿时间尽量控制在 200ms 左右。

注意，它是目标，不是保证。

G1 会根据这个目标调整每次回收多少 Region、年轻代大小等。

如果你把它设得太小，比如：

```bash
-XX:MaxGCPauseMillis=20
```

JVM 可能为了追求更短停顿而更频繁地 GC，吞吐量反而下降。

所以不要盲目把这个值调小。

常见起点：

```bash
-XX:MaxGCPauseMillis=200
```

然后结合 GC 日志和业务延迟再调。

### 6. G1 启动并发标记时机：-XX:InitiatingHeapOccupancyPercent

```bash
-XX:InitiatingHeapOccupancyPercent=45
```

这个参数常简称 IHOP。

它控制 G1 在堆占用达到多少比例时，开始并发标记周期。

`45` 大概意思是：

```text
老年代 / 整体堆占用到一定程度时，提前开始并发标记
```

如果触发太晚，可能来不及回收，导致 Full GC 风险上升。

如果触发太早，后台并发 GC 工作更多，CPU 成本会上升。

现代 G1 有自适应 IHOP，很多时候不需要手动改。只有 GC 日志显示并发周期经常来不及完成时，再考虑调整。

### 7. ZGC 参数

基础启用：

```bash
-XX:+UseZGC
```

Java 21 分代 ZGC：

```bash
-XX:+UseZGC -XX:+ZGenerational
```

ZGC 的核心目标是低延迟。一般不像 G1 那样频繁调一堆停顿参数。

使用 ZGC 时，我更关心这些问题：

- 堆是否给够。
- CPU 是否够。
- 分配速率是否太高。
- 是否出现 allocation stall。

如果堆太小，ZGC 也救不了。

### 8. 元空间：-XX:MetaspaceSize 和 -XX:MaxMetaspaceSize

Java 8 以后，永久代被元空间替代。

类元信息主要放在 Metaspace。

常见参数：

```bash
-XX:MetaspaceSize=256m
-XX:MaxMetaspaceSize=512m
```

| 参数 | 含义 |
| --- | --- |
| `MetaspaceSize` | 触发元空间 GC 的初始阈值，不是固定初始大小 |
| `MaxMetaspaceSize` | 元空间最大限制 |

如果项目大量使用动态代理、反射、热部署、脚本引擎，可能会遇到 Metaspace 增长问题。

但不要随便把 `MaxMetaspaceSize` 设得特别小，否则容易出现：

```text
java.lang.OutOfMemoryError: Metaspace
```

### 9. OOM 时导出堆快照

这组参数很实用：

```bash
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/data/logs/heapdump.hprof
```

含义：

| 参数 | 含义 |
| --- | --- |
| `HeapDumpOnOutOfMemoryError` | 发生 OOM 时自动导出堆快照 |
| `HeapDumpPath` | 指定 dump 文件保存路径 |

线上排查内存泄漏时，这比只看日志有用得多。

注意：dump 文件可能很大，要确保磁盘空间够，并且路径有写权限。

### 10. GC 日志

Java 9+ 推荐统一日志格式：

```bash
-Xlog:gc*:file=gc.log:time,uptime,level,tags
```

含义大概是：

```text
gc*       打印 GC 相关日志
gc.log    输出到 gc.log 文件
time      打印时间
uptime    打印 JVM 运行时长
level     打印日志级别
tags      打印日志标签
```

Java 8 常见写法：

```bash
-XX:+PrintGCDetails
-XX:+PrintGCDateStamps
-Xloggc:gc.log
```

如果是线上服务，建议至少打开 GC 日志。

没有 GC 日志，后面谈调优基本都是猜。

### 11. 一组比较常见的启动参数

普通 Java 17 / 21 后端服务，可以先从 G1 开始：

```bash
-Xms4g
-Xmx4g
-XX:+UseG1GC
-XX:MaxGCPauseMillis=200
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/data/logs/heapdump.hprof
-Xlog:gc*:file=/data/logs/gc.log:time,uptime,level,tags
```

如果是 Java 8：

```bash
-Xms4g
-Xmx4g
-XX:+UseG1GC
-XX:MaxGCPauseMillis=200
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/data/logs/heapdump.hprof
-XX:+PrintGCDetails
-XX:+PrintGCDateStamps
-Xloggc:/data/logs/gc.log
```

如果是低延迟服务，Java 21 可以评估：

```bash
-Xms8g
-Xmx8g
-XX:+UseZGC
-XX:+ZGenerational
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/data/logs/heapdump.hprof
-Xlog:gc*:file=/data/logs/gc.log:time,uptime,level,tags
```

这些不是“万能参数”。它们只是一个起点。

真正调优要看：

- GC 日志。
- 堆使用曲线。
- 分配速率。
- Young GC / Full GC 频率。
- P99 / P999 延迟。
- CPU 使用率。

## 十一、怎么选 GC？

我的建议：先不要调参，先选对方向。

### 普通后端服务

优先 G1。

```bash
-XX:+UseG1GC
-Xms4g -Xmx4g
```

JDK 9 以后默认就是 G1，大多数场景不用刻意换。

### 批处理 / 离线任务

可以考虑 Parallel GC。

```bash
-XX:+UseParallelGC
```

如果你更关心单位时间处理量，而不是单次停顿，Parallel 可能更合适。

### 低延迟服务

考虑 ZGC 或 Shenandoah。

```bash
-XX:+UseZGC
```

或者：

```bash
-XX:+UseShenandoahGC
```

前提是你真的有低延迟需求，并且已经通过 GC 日志或监控确认 GC 是问题来源。

### 小工具 / 小内存应用

Serial GC 不丢人。

```bash
-XX:+UseSerialGC
```

小堆应用里，简单可能就是优势。

## 十二、对比表

| GC | 主要版本脉络 | 核心目标 | 特点 | 适合场景 | 不适合场景 |
| --- | --- | --- | --- | --- | --- |
| Serial | 早期一直存在 | 简单 | 单线程、STW、实现简单 | 小堆、小工具、单核环境 | 大型服务端应用 |
| Parallel | Java 8 常见默认 | 吞吐量 | 多线程 STW，回收速度快 | 批处理、离线任务、吞吐优先 | 延迟敏感服务 |
| CMS | Java 8 常见低延迟选择，后续淘汰 | 降低老年代停顿 | 并发标记清除，不压缩，易碎片 | 老 Java 8 低延迟项目 | 新项目，不建议继续选 |
| G1 | JDK 9 默认 | 平衡吞吐和停顿 | Region、可预测停顿目标、混合回收 | 大多数后端服务 | 极端低延迟场景 |
| Epsilon | JDK 11 | 无回收 | 只分配不回收，堆满 OOM | 基准测试、短命任务 | 长期运行服务 |
| ZGC | JDK 11 实验，JDK 15 正式 | 低延迟 | 大部分并发执行，短暂停顿，大堆友好 | 低延迟、大堆服务 | CPU 紧张、吞吐极限优先 |
| Shenandoah | JDK 12 左右进入主线，视发行版支持 | 低延迟 | 并发标记、并发整理 | 低延迟服务，大堆应用 | 发行版不支持或团队经验不足 |
| Generational ZGC | JDK 21 重要升级 | 分代低延迟 | ZGC + 分代假设，减少不必要扫描 | Java 21+ 低延迟服务 | 旧 JDK 或不需要低延迟的普通服务 |

## 十三、最后怎么理解 GC 演进？

可以按这个脉络记：

```text
Serial      简单，小堆够用
Parallel    多线程 STW，追求吞吐
CMS         尝试降低老年代停顿，但碎片问题明显
G1          Region 化，成为服务端默认选择
ZGC/Shenandoah  低延迟，大部分工作并发化
Generational ZGC  低延迟 GC 重新引入分代优势
```

GC 的演进不是“新的永远替代旧的”。

它更像是在不同目标之间做选择：

```text
吞吐量
停顿时间
内存占用
CPU 成本
实现复杂度
```

调 GC 前，先看 GC 日志和监控。没有数据，只凭感觉换 GC，通常是在制造新问题。
