## MySQL字符集与排序规则怎么选

刚学 MySQL 字符集时，我一直把下面两行混在一起看：

```sql
CHARACTER SET utf8mb4
COLLATE utf8mb4_general_ci
```

后来才发现，这里其实有两个问题。

`CHARACTER SET` 管的是能不能存。比如中文、英文、emoji、生僻字，到底能不能正常落进库里。

`COLLATE` 管的是怎么比。比如 `a` 和 `A` 算不算同一个值，`ORDER BY` 排出来的顺序是什么，唯一索引碰到两个看起来差不多的字符串时该不该报冲突。

所以这篇先不急着背配置。先把 MySQL 这些字符集是怎么一路演化过来的说清楚，再看项目里到底怎么选。

### 先说老坑：latin1

很多老版本 MySQL 的默认字符集是 `latin1`，默认排序规则常见的是 `latin1_swedish_ci`。

如果项目只处理英文，问题可能不明显。到了中文项目，这套默认值就容易闹心：插入中文乱码，导 SQL 文件乱码，应用连接没配好也乱码。更麻烦的是，有些库后来被人手动改过，于是同一个项目里可能同时出现 `latin1`、`utf8`、`utf8mb4`。

这类历史库最怕一句话：“先跑起来再说。”

能跑是一回事，几年后排查乱码又是另一回事。

所以以前很多中文项目建库时会主动写：

```sql
CREATE DATABASE demo DEFAULT CHARACTER SET utf8;
```

这比 `latin1` 好很多，但 `utf8` 这个名字在 MySQL 里也不算省心。

### MySQL里的utf8不是完整UTF-8

平时我们说 UTF-8，通常默认它能表示完整 Unicode 字符。

MySQL 里的 `utf8` 长期不是这个意思。它实际是 `utf8mb3` 的别名，最多用 3 个字节存一个字符。普通中文、英文基本没事，但 emoji、部分生僻字和一些扩展符号需要 4 个字节。

这就是为什么有些系统平时用着挺正常，用户昵称里一放表情就炸。

常见错误大概长这样：

```text
Incorrect string value
```

这类问题很烦，因为它不像 SQL 语法错那样一眼就能定位。表结构、连接参数、应用日志、导入文件编码，都可能参与制造现场。

现在再建新库，我一般不会写 `utf8`，直接写：

```sql
CHARACTER SET utf8mb4
```

别省那几个字符。

### utf8mb4是今天的默认答案

`utf8mb4` 可以理解为 MySQL 里更完整的 UTF-8 实现。它最多用 4 个字节存一个字符，能覆盖 BMP 字符，也能覆盖 BMP 之外的补充字符。

说人话就是：中文、英文、emoji、生僻字，先别在存储层给自己挖坑。

MySQL 8 里，默认字符集已经是 `utf8mb4`，默认排序规则是 `utf8mb4_0900_ai_ci`。这个默认值本身就挺有参考意义。新项目如果没有非常特殊的历史包袱，字符集就选 `utf8mb4`。

真正需要多想一步的是排序规则。

### 排序规则的名字怎么读

先看你提到的这个：

```text
utf8mb4_general_ci
```

它可以拆成三段：

- `utf8mb4`：它属于 `utf8mb4` 这个字符集；
- `general`：通用排序规则；
- `ci`：case-insensitive，不区分大小写。

所以在这个规则下，`abc` 和 `ABC` 做比较时通常会被认为相等：

```sql
SELECT 'abc' COLLATE utf8mb4_general_ci = 'ABC';
```

再看 MySQL 8 默认的：

```text
utf8mb4_0900_ai_ci
```

这里的 `0900` 指 Unicode Collation Algorithm 9.0.0；`ai` 是 accent-insensitive，不区分重音；`ci` 还是不区分大小写。

常见后缀不用死记，先记几个够用的：

- `_ci`：不区分大小写；
- `_cs`：区分大小写；
- `_ai`：不区分重音；
- `_as`：区分重音；
- `_bin`：按更接近编码值的方式比较，适合精确匹配。

排序规则不是只影响 `ORDER BY`。它还会影响 `where name = ?`、唯一索引判重、字符串函数比较。这个点很容易被低估。

### utf8mb4_general_ci到底怎么样

`utf8mb4_general_ci` 在老项目里很常见。

它不是不能用。它的优点也很清楚：支持 `utf8mb4`，不区分大小写，兼容性好，MySQL 5.7 和 MySQL 8 都支持。很多普通业务字段用它也没什么大问题。

但它的排序和比较规则比较粗，处理一些语言字符时不如 Unicode 规则准确。它还有一个老规则的特点：`PAD SPACE`，尾部空格的比较行为和 MySQL 8 的 `0900` 系列不完全一样。

所以我的看法是：老项目已经统一用了 `utf8mb4_general_ci`，可以先保持。新项目就没必要特意选它了。

尤其别为了“升级感”直接全库改排序规则。排序规则一改，唯一索引、排序结果、字符串比较都可能变化。真要改，按数据迁移处理，不要当成普通配置项。

### MySQL 5.7怎么选

如果项目还要兼容 MySQL 5.7，不能用 `utf8mb4_0900_ai_ci`。

我会这样建库：

```sql
CREATE DATABASE demo
  DEFAULT CHARACTER SET utf8mb4
  DEFAULT COLLATE utf8mb4_unicode_ci;
```

`utf8mb4_unicode_ci` 比 `utf8mb4_general_ci` 更接近 Unicode 排序规则，语义上稳一点。普通中文业务系统用它足够。

如果你接手的是老库，已经全部是 `utf8mb4_general_ci`，那就先别动。保持全库一致，通常比混用多个排序规则更重要。

### MySQL 8怎么选

MySQL 8 新项目，我会直接用这个：

```sql
CREATE DATABASE demo
  DEFAULT CHARACTER SET utf8mb4
  DEFAULT COLLATE utf8mb4_0900_ai_ci;
```

这个选择没什么花活。它是 MySQL 8 的默认 `utf8mb4` 排序规则，基于更新的 Unicode 9.0.0 规则，不区分大小写，也不区分重音。

中文为主的后台系统、博客、论坛、电商、评论系统，用它基本够了。我们更多关心的是别乱码、别因为 emoji 写不进去、查询比较行为别太反直觉。

### 有些字段不要跟着全库默认走

数据库默认排序规则适合大部分字段，但不是所有字段都该跟着默认值。

比如用户名要不要区分 `Tom` 和 `tom`，这个要看产品规则。如果要区分，可以在字段上单独指定：

```sql
CREATE TABLE user_account (
  id BIGINT PRIMARY KEY,
  username VARCHAR(64)
    CHARACTER SET utf8mb4
    COLLATE utf8mb4_0900_as_cs
);
```

`as_cs` 表示区分重音、区分大小写。

再比如 token、hash、验证码这类字段，通常就应该精确比较。`abc` 和 `ABC` 不该被看成一样。

可以用 `_bin`：

```sql
CREATE TABLE access_token (
  id BIGINT PRIMARY KEY,
  token VARCHAR(128)
    CHARACTER SET utf8mb4
    COLLATE utf8mb4_bin
);
```

如果它本质上就是二进制内容，比如 hash 摘要，也可以直接考虑 `BINARY`、`VARBINARY`，不用硬塞进字符串类型。

### 数据库默认值只影响以后

创建数据库时写的字符集和排序规则，是这个库的默认值：

```sql
CREATE DATABASE demo
  DEFAULT CHARACTER SET utf8mb4
  DEFAULT COLLATE utf8mb4_0900_ai_ci;
```

后续创建表时，如果表没有单独指定，就继承数据库默认值。创建字段时，如果字段也没有单独指定，就继承表默认值。

大概是这个顺序：

```text
服务器默认值 -> 数据库默认值 -> 表默认值 -> 字段定义
```

越往后越具体，也越优先。

这里有个容易误会的点：你改了数据库默认字符集，不代表已有表、已有字段自动改了。老库迁移时一定要查清楚当前状态。

查看数据库默认值：

```sql
SELECT DEFAULT_CHARACTER_SET_NAME, DEFAULT_COLLATION_NAME
FROM INFORMATION_SCHEMA.SCHEMATA
WHERE SCHEMA_NAME = 'demo';
```

查看当前连接相关配置：

```sql
SHOW VARIABLES LIKE 'character_set%';
SHOW VARIABLES LIKE 'collation%';
```

查看 `utf8mb4` 有哪些排序规则：

```sql
SHOW COLLATION WHERE Charset = 'utf8mb4';
```

### 连接层也别漏

表字段是 `utf8mb4`，只能说明存储层没故意为难你。应用连接 MySQL 时，连接层也要使用正确的字符集。

SQL 层可以这样指定当前连接：

```sql
SET NAMES utf8mb4;
```

实际项目里更多是靠驱动和连接池配置。新版本驱动通常处理得还不错，但老项目最好检查一下。乱码问题最烦的一点是，它不一定马上报错，可能是数据写进去以后才发现不对。

尤其是导入 SQL 文件、从老库迁移、换驱动版本时，别只看表结构。

### 我的选择习惯

如果让我从头建一个库，我会按这个顺序选：

MySQL 8 及以上：

```sql
CREATE DATABASE demo
  DEFAULT CHARACTER SET utf8mb4
  DEFAULT COLLATE utf8mb4_0900_ai_ci;
```

MySQL 5.7：

```sql
CREATE DATABASE demo
  DEFAULT CHARACTER SET utf8mb4
  DEFAULT COLLATE utf8mb4_unicode_ci;
```

老项目已经统一使用 `utf8mb4_general_ci`：先保持，除非已经确认它带来了具体问题。

特殊字段单独看：

- 普通名称、标题、备注，用默认 `_ci` 通常可以；
- 用户名、业务编码，如果规则要求大小写敏感，单独用 `_as_cs`；
- token、hash、验证码，考虑 `_bin` 或二进制类型；
- 多语言排序特别严格，再看语言专用 collation。

我现在会把结论压成一句话：字符集先选 `utf8mb4`，排序规则再按 MySQL 版本和字段比较需求定。
