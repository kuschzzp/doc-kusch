# DDD 真实项目设计教程：从业务流程到代码落地

很多人第一次接触 DDD，也就是 Domain-Driven Design，领域驱动设计，都会有一种感觉：

> 概念好像都听过：实体、值对象、聚合、领域事件、仓储、限界上下文。  
> 但一到真实项目，就不知道第一步该怎么下手。

这很正常。

因为 DDD 不是一种固定目录结构，也不是“把 Service 拆得更复杂”。它真正解决的是一个问题：

> 当业务越来越复杂时，代码如何仍然能够清楚地表达业务？

这篇文章不从术语开始，而是从真实项目设计的角度讲：拿到一个需求，如何一步一步用 DDD 思维拆解，最后落到代码结构上。

---

## 一、先纠正一个常见误区

很多项目的设计流程是这样的：

```text
先设计数据库表
再根据表生成 Entity
然后写 Service
最后暴露 Controller
```

这种方式适合简单 CRUD。

比如用户管理、字典配置、菜单管理、文章分类，这类业务主要就是增删改查，按数据库表建模没有太大问题。

但如果系统开始出现这些情况，就会越来越痛苦：

```text
订单状态很多
支付、退款、库存、物流互相影响
一个操作会触发多个后续动作
同一个概念在不同模块里含义不同
Service 里有大量 if else
业务规则散落在 Controller、Service、SQL、定时任务里
新人看表结构知道字段含义，却看不懂业务含义
```

这时，如果还以数据库为中心设计，代码很容易变成：

```text
OrderService
  createOrder()
  payOrder()
  cancelOrder()
  refundOrder()
  lockStock()
  releaseStock()
  sendMessage()
  updateCoupon()
  addPoints()
```

最后 `OrderService` 什么都管，变成一个巨大的业务垃圾桶。

DDD 的设计顺序不一样。

它更推荐这样：

```text
业务流程 -> 业务规则 -> 领域边界 -> 聚合模型 -> 应用服务 -> 持久化设计
```

也就是说，先理解业务，再设计模型，最后才考虑数据库和框架。

---

## 二、DDD 真实项目设计的核心流程

在真实项目中，可以按照下面 8 个步骤来做 DDD 设计。

```text
1. 梳理业务场景
2. 找出业务动作
3. 提炼业务规则
4. 划分限界上下文
5. 识别实体和值对象
6. 设计聚合和聚合根
7. 设计应用服务和领域事件
8. 最后设计数据库和接口
```

下面我们用具体案例展开。

---

## 三、案例一：电商订单系统

假设我们要设计一个电商订单系统。

业务方给出的需求大概是：

```text
用户可以选择商品下单
下单时需要锁定库存
用户需要在 30 分钟内完成支付
支付成功后订单进入待发货状态
商家发货后订单进入待收货状态
用户可以取消未发货订单
已支付订单取消后需要退款
订单超时未支付会自动关闭
订单关闭后需要释放库存
```

如果按照传统方式，很容易先设计表：

```text
user
product
order
order_item
payment
refund
stock
```

但 DDD 不建议第一步就建表。

第一步应该问：

> 这个系统里有哪些业务动作？

---

## 四、第一步：梳理业务动作

从需求中可以提取出这些动作：

```text
创建订单
提交订单
锁定库存
支付订单
取消订单
关闭超时订单
商家发货
确认收货
申请退款
退款成功
```

这些动作通常可以对应成 Command，也就是命令。

命令表示“用户或系统想做什么”：

```text
CreateOrder
SubmitOrder
LockStock
PayOrder
CancelOrder
ExpireOrder
ShipOrder
ConfirmReceipt
ApplyRefund
SucceedRefund
```

与此同时，还可以提炼出事件。

事件表示“已经发生了什么”：

```text
OrderCreated
OrderSubmitted
StockLocked
OrderPaid
OrderCancelled
OrderExpired
OrderShipped
OrderReceived
RefundApplied
RefundSucceeded
```

命令和事件的区别非常重要：

```text
PayOrder 是请求：我要支付订单
OrderPaid 是事实：订单已经支付成功

CancelOrder 是请求：我要取消订单
OrderCancelled 是事实：订单已经取消
```

在真实项目中，只要你能把命令和事件列清楚，业务流程就已经清晰了一半。

---

## 五、第二步：提炼业务规则

接下来继续问：

> 每个动作在什么条件下允许发生？

比如取消订单。

业务规则可能是：

```text
待支付订单可以取消
已支付但未发货订单可以取消，但需要退款
已发货订单不能直接取消，只能申请售后
已完成订单不能取消
已取消订单不能重复取消
订单取消后需要释放库存
```

再比如支付订单：

```text
只有待支付订单可以支付
已取消订单不能支付
已超时订单不能支付
支付金额必须等于订单应付金额
支付成功后订单状态变为待发货
```

这些规则就是领域规则。

DDD 的重点就是给这些规则找到合适的位置，而不是让它们散落在各个接口里。

如果你在 Controller 里写：

```ts
if (order.status === 'SHIPPED') {
  throw new Error('已发货订单不能取消')
}
```

在定时任务里又写一遍：

```ts
if (order.status !== 'PENDING_PAYMENT') {
  return
}
```

在退款流程里再写一遍：

```ts
if (order.status === 'CANCELLED') {
  throw new Error('订单已取消')
}
```

时间一长，规则就会分散、重复、互相冲突。

更好的做法是把订单状态变化规则放回 `Order` 这个领域对象里。

---

## 六、第三步：划分限界上下文

接下来进入 DDD 里最重要的一步：划分限界上下文。

限界上下文可以理解成：

> 一个业务概念在某个边界内有明确、稳定的含义。

电商系统里，不应该只有一个巨大的“订单模块”。至少可以拆成这些上下文：

```text
商品上下文 Catalog
负责：商品资料、SKU、价格、上下架

订单上下文 Ordering
负责：订单创建、订单明细、订单状态流转、订单金额快照

库存上下文 Inventory
负责：库存数量、库存锁定、库存扣减、库存释放

支付上下文 Payment
负责：支付单、支付流水、支付回调、退款单

物流上下文 Shipping
负责：发货、运单、签收

营销上下文 Promotion
负责：优惠券、活动、满减、折扣
```

注意，限界上下文不是数据库表，也不一定等于微服务。

即使是单体项目，也可以按照限界上下文拆模块：

```text
src/
  catalog/
  ordering/
  inventory/
  payment/
  shipping/
  promotion/
```

以后要不要拆成微服务，是部署层面的选择。DDD 首先关心的是业务边界。

---

## 七、为什么不能共用一个全局模型？

很多项目喜欢定义一个全局 `Product`：

```ts
class Product {
  id: string
  name: string
  price: number
  stock: number
  categoryId: string
  status: string
}
```

然后订单、库存、营销、搜索全部依赖这个 `Product`。

一开始很方便，但业务复杂后会出问题。

因为不同上下文对“商品”的关注点不同：

```text
商品上下文关心：商品名称、类目、上下架、详情、SKU
订单上下文关心：下单时的商品名称、下单价、SKU 快照
库存上下文关心：SKU 当前库存、锁定库存、可售库存
营销上下文关心：商品是否参与活动、折扣规则
搜索上下文关心：标题、标签、排序权重
```

订单上下文不应该依赖完整的商品模型。

它只需要一个商品快照：

```ts
type ProductSnapshot = {
  skuId: string
  name: string
  unitPrice: Money
}
```

为什么是快照？

因为订单里的商品名称和价格应该是下单那一刻的事实。商品系统后面改价，不应该影响历史订单。

这就是 DDD 里非常重要的思维：

> 同一个词，在不同上下文里可以有不同模型。

---

## 八、第四步：识别实体和值对象

DDD 里有两个基础概念：实体和值对象。

实体有唯一身份，会随着时间变化。

例如：

```text
Order
Payment
Refund
Shipment
Customer
Product
```

值对象没有独立身份，只由属性值决定。

例如：

```text
Money
Address
PhoneNumber
Email
OrderNo
ProductSnapshot
DateRange
```

比如金额 `Money` 就适合作为值对象：

```ts
class Money {
  constructor(
    public readonly amount: number,
    public readonly currency: string
  ) {
    if (amount < 0) {
      throw new Error('金额不能小于 0')
    }

    if (!currency) {
      throw new Error('币种不能为空')
    }
  }

  add(other: Money): Money {
    if (this.currency !== other.currency) {
      throw new Error('币种不一致')
    }

    return new Money(this.amount + other.amount, this.currency)
  }

  equals(other: Money): boolean {
    return this.amount === other.amount && this.currency === other.currency
  }
}
```

这样做的好处是：

```text
金额不能小于 0
不同币种不能直接相加
金额比较有统一逻辑
```

这些规则不需要到处写。

---

## 九、第五步：设计聚合和聚合根

聚合是 DDD 中最容易让人困惑的概念。

可以先用一句话理解：

> 聚合是一组必须一起保持一致的领域对象，聚合根是外部访问这组对象的入口。

在订单上下文中，通常可以这样设计：

```text
Order 是聚合根
OrderItem 是聚合内部对象
```

外部不应该直接修改 `OrderItem`，而应该通过 `Order`：

```ts
order.addItem(productSnapshot, quantity)
order.submit()
order.pay(paymentId)
order.cancel(reason)
order.ship(trackingNo)
```

这样做的原因是：订单明细的变化会影响订单总金额，订单状态也会限制订单明细是否可修改。

如果外部可以直接改 `OrderItem`，就很容易破坏订单一致性。

例如：

```ts
class Order {
  private items: OrderItem[] = []
  private events: DomainEvent[] = []

  constructor(
    public readonly id: string,
    public readonly customerId: string,
    private status: OrderStatus = OrderStatus.PendingPayment
  ) {}

  addItem(product: ProductSnapshot, quantity: number) {
    if (this.status !== OrderStatus.PendingPayment) {
      throw new Error('只有待支付订单可以修改商品')
    }

    if (quantity <= 0) {
      throw new Error('商品数量必须大于 0')
    }

    this.items.push(new OrderItem(product, quantity))
  }

  submit() {
    if (this.items.length === 0) {
      throw new Error('不能提交空订单')
    }

    if (this.status !== OrderStatus.PendingPayment) {
      throw new Error('当前订单状态不能提交')
    }

    this.events.push(new OrderSubmitted(this.id))
  }

  pay(paymentId: string, paidAmount: Money) {
    if (this.status !== OrderStatus.PendingPayment) {
      throw new Error('只有待支付订单可以支付')
    }

    if (!paidAmount.equals(this.totalAmount())) {
      throw new Error('支付金额与订单金额不一致')
    }

    this.status = OrderStatus.Paid
    this.events.push(new OrderPaid(this.id, paymentId))
  }

  cancel(reason: string) {
    if (this.status === OrderStatus.Shipped) {
      throw new Error('已发货订单不能直接取消')
    }

    if (this.status === OrderStatus.Completed) {
      throw new Error('已完成订单不能取消')
    }

    if (this.status === OrderStatus.Cancelled) {
      throw new Error('订单已经取消')
    }

    this.status = OrderStatus.Cancelled
    this.events.push(new OrderCancelled(this.id, reason))
  }

  totalAmount(): Money {
    return this.items.reduce(
      (total, item) => total.add(item.subtotal()),
      new Money(0, 'CNY')
    )
  }

  pullEvents(): DomainEvent[] {
    const currentEvents = [...this.events]
    this.events = []
    return currentEvents
  }
}
```

这样一来，订单的核心规则都集中在 `Order` 内部。

应用层不需要到处判断订单状态，只需要调用：

```ts
order.cancel(reason)
```

这就是 DDD 的一个重要目标：

> 代码读起来像业务语言，而不是像数据库操作脚本。

## 十、聚合边界怎么判断？

判断两个对象是否应该放在同一个聚合里，可以问几个问题：

```text
1. 它们是否必须在同一次操作里保持强一致？
2. 外部是否应该只能通过某个根对象访问它们？
3. 子对象离开根对象后是否没有意义？
4. 修改一个对象时，是否必须同时校验另一个对象？
```

如果答案是“是”，它们可能属于同一个聚合。

例如：

```text
Order 和 OrderItem
```

订单项离开订单没有独立意义，修改订单项会影响订单金额，所以它们适合放在一个聚合里。

但下面这个关系就不适合放在一个聚合里：

```text
Customer 和 Order
```

一个客户可能有成千上万个订单。如果把所有订单都放进客户聚合，每次加载客户都可能加载大量订单，聚合会变得非常大。

更合理的是：

```text
Customer 是一个聚合
Order 是另一个聚合
Order 里保存 customerId
```

聚合设计有一个实用原则：

> 聚合尽量小，只保护真正需要强一致的业务规则。

---

## 十一、第六步：设计应用服务

应用服务负责流程编排，但不应该承载大量业务规则。

比如取消订单的应用服务：

```ts
class CancelOrderUseCase {
  constructor(
    private readonly orderRepository: OrderRepository,
    private readonly eventBus: EventBus
  ) {}

  async execute(command: CancelOrderCommand) {
    const order = await this.orderRepository.findById(command.orderId)

    if (!order) {
      throw new Error('订单不存在')
    }

    order.cancel(command.reason)

    await this.orderRepository.save(order)

    await this.eventBus.publish(order.pullEvents())
  }
}
```

应用服务做这些事情：

```text
加载聚合
调用领域方法
保存聚合
发布领域事件
控制事务
调用外部上下文接口
```

但这类规则不应该主要写在应用服务里：

```text
已发货订单不能取消
支付金额必须等于订单金额
空订单不能提交
已完成订单不能取消
```

这些是领域规则，应该放在领域模型里。

---

## 十二、第七步：使用领域事件做跨上下文协作

订单取消后，需要释放库存，也可能需要退款。

很多项目会在 `OrderService.cancelOrder()` 里这样写：

```ts
await orderRepository.save(order)
await stockService.release(order.id)
await paymentService.refund(order.id)
await messageService.send(...)
```

这样写短期能跑，但长期会让订单服务和库存、支付、消息强耦合。

DDD 更推荐用领域事件表达业务事实：

```text
OrderCancelled
```

然后让其他上下文订阅这个事件：

```text
Inventory 监听 OrderCancelled -> 释放库存
Payment 监听 OrderCancelled -> 如果已支付，创建退款
Notification 监听 OrderCancelled -> 发送通知
```

订单上下文只负责说明：

```text
订单已经取消
```

至于库存怎么释放、支付怎么退款，是其他上下文自己的职责。

这样可以降低上下文之间的耦合。

---

## 十三、跨上下文调用要加“反腐层”

订单创建时，需要知道商品名称、价格、是否可售。

错误做法是让订单上下文直接依赖商品上下文的 `Product`：

```ts
import { Product } from '@/catalog/domain/Product'
```

这会导致订单模型被商品模型污染。

更好的做法是在订单上下文里定义自己需要的接口：

```ts
interface CatalogClient {
  getProductSnapshot(skuId: string): Promise<ProductSnapshot>
}
```

订单上下文只认识自己的 `ProductSnapshot`：

```ts
type ProductSnapshot = {
  skuId: string
  name: string
  unitPrice: Money
  available: boolean
}
```

具体怎么调用商品系统，可以放在基础设施层：

```ts
class HttpCatalogClient implements CatalogClient {
  async getProductSnapshot(skuId: string): Promise<ProductSnapshot> {
    const response = await http.get(`/catalog/skus/${skuId}`)

    return {
      skuId: response.data.id,
      name: response.data.title,
      unitPrice: new Money(response.data.priceCent, 'CNY'),
      available: response.data.status === 'ON_SALE'
    }
  }
}
```

这层转换就是反腐层。

它的作用是：

> 把外部上下文的数据结构，翻译成当前上下文能理解的模型。

---

## 十四、第八步：最后设计数据库

DDD 不是不要数据库，而是不让数据库成为业务设计的起点。

比如订单聚合可以落库成：

```text
orders
order_items
```

支付上下文可以有：

```text
payments
payment_transactions
refunds
```

库存上下文可以有：

```text
inventory_items
inventory_locks
inventory_records
```

领域模型和数据库表不一定一一对应。

Repository 的职责是：

```text
把数据库记录还原成领域聚合
把领域聚合保存回数据库
隐藏 ORM、SQL、缓存等技术细节
```

例如：

```ts
interface OrderRepository {
  findById(orderId: string): Promise<Order | null>
  save(order: Order): Promise<void>
}
```

领域层只依赖这个接口，不关心底层是 MySQL、PostgreSQL、MongoDB，还是内存实现。

---

## 十五、一个推荐的代码结构

在单体项目里，可以这样组织：

```text
src/
  ordering/
    domain/
      Order.ts
      OrderItem.ts
      OrderStatus.ts
      Money.ts
      ProductSnapshot.ts
      events/
        OrderPaid.ts
        OrderCancelled.ts
    application/
      CreateOrderUseCase.ts
      PayOrderUseCase.ts
      CancelOrderUseCase.ts
    infrastructure/
      OrderRepositoryImpl.ts
      HttpCatalogClient.ts
      OrderOrmEntity.ts
    interfaces/
      OrderController.ts

  inventory/
    domain/
      InventoryItem.ts
      StockReservation.ts
    application/
      ReserveStockUseCase.ts
      ReleaseStockUseCase.ts
    infrastructure/
    interfaces/

  payment/
    domain/
      Payment.ts
      Refund.ts
    application/
      CreatePaymentUseCase.ts
      HandlePaymentCallbackUseCase.ts
      CreateRefundUseCase.ts
    infrastructure/
    interfaces/
```

依赖方向应该是：

```text
interfaces -> application -> domain
infrastructure -> domain
infrastructure -> application
```

但是：

```text
domain 不能依赖 infrastructure
domain 不能依赖 controller
domain 不能依赖 ORM
domain 不能依赖 HTTP
domain 不能依赖具体数据库
```

领域层应该尽量保持纯粹。

## 十六、案例二：在线教育报名系统

再换一个场景，帮助理解。

假设我们要做一个在线教育报名系统。

业务需求：

```text
学员可以报名课程
课程有班级名额限制
报名后需要 15 分钟内支付
支付成功后报名成功
超时未支付自动关闭报名单并释放名额
开课前可以取消报名并退款
开课后取消报名需要人工审核
课程完成后不能退款
```

如果直接建表，可能是：

```text
student
course
class
enrollment
payment
refund
```

但用 DDD 思维，先拆业务上下文。

## 十七、在线教育系统的限界上下文

可以拆成：

```text
课程上下文 Course
负责：课程名称、课程介绍、价格、上下架

班级名额上下文 Capacity
负责：班级容量、锁定名额、释放名额、确认占用

报名上下文 Enrollment
负责：报名单、报名状态、报名规则

支付上下文 Payment
负责：支付单、支付回调、退款

学习上下文 Learning
负责：课时、学习进度、作业、结课
```

这里最重要的是：

> 报名不是课程，报名也不是支付，报名是学员参加课程的一次业务关系。

所以报名上下文里的核心模型应该是：

```text
Enrollment
```

它可能包含：

```text
enrollmentId
studentId
courseSnapshot
seatReservationId
price
status
createdAt
paidAt
```

其中 `courseSnapshot` 是课程快照，不是课程实体。

## 十八、报名聚合示例

报名状态可以是：

```text
PendingPayment 待支付
Paid 已支付
Cancelled 已取消
Expired 已超时关闭
Refunding 退款中
Refunded 已退款
Completed 已完成
```

报名聚合可以这样写：

```ts
class Enrollment {
  private events: DomainEvent[] = []

  constructor(
    public readonly id: string,
    public readonly studentId: string,
    public readonly courseSnapshot: CourseSnapshot,
    public readonly seatReservationId: string,
    private status: EnrollmentStatus
  ) {}

  pay(paymentId: string) {
    if (this.status !== EnrollmentStatus.PendingPayment) {
      throw new Error('只有待支付报名单可以支付')
    }

    this.status = EnrollmentStatus.Paid
    this.events.push(new EnrollmentPaid(this.id, paymentId))
  }

  cancel(reason: string, courseStarted: boolean) {
    if (this.status === EnrollmentStatus.Completed) {
      throw new Error('已完成课程不能取消报名')
    }

    if (this.status === EnrollmentStatus.Refunded) {
      throw new Error('已退款报名单不能重复取消')
    }

    if (courseStarted) {
      this.status = EnrollmentStatus.Refunding
      this.events.push(new EnrollmentRefundReviewRequired(this.id, reason))
      return
    }

    this.status = EnrollmentStatus.Cancelled
    this.events.push(new EnrollmentCancelled(this.id, reason))
  }

  expire() {
    if (this.status !== EnrollmentStatus.PendingPayment) {
      throw new Error('只有待支付报名单可以超时关闭')
    }

    this.status = EnrollmentStatus.Expired
    this.events.push(new EnrollmentExpired(this.id, this.seatReservationId))
  }

  complete() {
    if (this.status !== EnrollmentStatus.Paid) {
      throw new Error('只有已支付报名单可以完成课程')
    }

    this.status = EnrollmentStatus.Completed
    this.events.push(new EnrollmentCompleted(this.id))
  }

  pullEvents(): DomainEvent[] {
    const currentEvents = [...this.events]
    this.events = []
    return currentEvents
  }
}
```

这里的关键不是代码形式，而是职责划分：

```text
Enrollment 只负责报名单自己的状态变化
它不直接扣名额
它不直接调支付渠道
它不直接创建学习进度
```

这些事情通过领域事件交给其他上下文处理。

## 十九、名额为什么不放进报名聚合？

很多人会想：报名成功就占一个名额，那名额是不是应该放在 `Enrollment` 里？

通常不建议。

因为名额有自己的业务规则：

```text
班级总名额是多少
当前已确认名额是多少
当前锁定名额是多少
同一个学员能不能重复锁定
锁定多久过期
名额不足时如何处理
```

这些规则属于 Capacity 上下文，而不是 Enrollment 上下文。

可以设计一个 `ClassCapacity` 聚合：

```ts
class ClassCapacity {
  constructor(
    public readonly classId: string,
    private totalSeats: number,
    private confirmedSeats: number,
    private reservedSeats: number
  ) {}

  reserveSeat(studentId: string): SeatReservation {
    if (this.availableSeats() <= 0) {
      throw new Error('班级名额不足')
    }

    this.reservedSeats += 1

    return new SeatReservation(
      generateId(),
      this.classId,
      studentId,
      new Date(Date.now() + 15 * 60 * 1000)
    )
  }

  confirmReservation(reservation: SeatReservation) {
    if (reservation.isExpired()) {
      throw new Error('名额锁定已过期')
    }

    this.reservedSeats -= 1
    this.confirmedSeats += 1
  }

  releaseReservation() {
    if (this.reservedSeats <= 0) {
      return
    }

    this.reservedSeats -= 1
  }

  private availableSeats(): number {
    return this.totalSeats - this.confirmedSeats - this.reservedSeats
  }
}
```

这样就很清楚：

```text
Enrollment 关心报名状态
Capacity 关心名额变化
Payment 关心支付和退款
Learning 关心学习进度
```

## 二十、报名流程如何编排？

创建报名单的应用服务可以这样：

```ts
class CreateEnrollmentUseCase {
  constructor(
    private readonly courseClient: CourseClient,
    private readonly capacityService: CapacityService,
    private readonly enrollmentRepository: EnrollmentRepository
  ) {}

  async execute(command: CreateEnrollmentCommand) {
    const course = await this.courseClient.getCourseSnapshot(command.courseId)

    if (!course.canEnroll) {
      throw new Error('课程不可报名')
    }

    const reservation = await this.capacityService.reserveSeat({
      classId: command.classId,
      studentId: command.studentId
    })

    const enrollment = Enrollment.create({
      studentId: command.studentId,
      courseSnapshot: course,
      seatReservationId: reservation.id,
      price: course.price
    })

    await this.enrollmentRepository.save(enrollment)

    return enrollment.id
  }
}
```

支付成功后：

```text
PaymentSucceeded
  -> Enrollment 标记为已支付
  -> 发布 EnrollmentPaid
  -> Capacity 确认名额
  -> Learning 创建学习记录
```

超时未支付：

```text
ExpireEnrollment
  -> Enrollment 标记为已超时
  -> 发布 EnrollmentExpired
  -> Capacity 释放名额
```

这个设计让每个上下文都只处理自己的规则。

## 二十一、哪些规则放在哪里？

真实项目里，经常纠结“这段逻辑应该放在哪里”。

可以用下面的判断方式。

### 放在实体或聚合里

如果规则和对象自身状态强相关，就放在实体或聚合里。

例如：

```text
订单已发货不能取消
空订单不能提交
报名单超时后不能支付
已完成课程不能退款
```

这些应该放在：

```text
Order.cancel()
Order.submit()
Enrollment.pay()
Enrollment.cancel()
```

### 放在值对象里

如果规则和某个值本身有关，就放在值对象里。

例如：

```text
金额不能小于 0
手机号格式必须正确
时间范围的开始时间不能晚于结束时间
不同币种不能相加
```

这些应该放在：

```text
Money
PhoneNumber
DateRange
```

### 放在领域服务里

如果规则是纯业务规则，但不自然属于某一个实体，可以放在领域服务里。

例如：

```text
根据会员等级、优惠券、活动计算最终价格
根据多个规则判断是否命中风控
根据多个实体计算排课冲突
```

领域服务不应该依赖数据库、HTTP、消息队列。

### 放在应用服务里

如果逻辑是流程编排，就放在应用服务里。

例如：

```text
先查订单
再调用 order.cancel()
然后保存订单
最后发布事件
```

应用服务可以调用仓储、外部接口、消息总线，但尽量不要写复杂业务判断。

### 放在基础设施层

如果是技术细节，就放在基础设施层。

例如：

```text
SQL 查询
ORM 映射
HTTP 调用
Redis 缓存
消息队列发送
第三方支付 SDK
```

## 二十二、DDD 和事务怎么配合？

一个实用原则是：

> 一个聚合内部尽量使用强一致，多个聚合之间尽量使用最终一致。

例如订单和订单项属于一个聚合：

```text
创建订单时，订单和订单项应该在一个事务里保存
```

但订单和库存属于不同上下文：

```text
订单取消后释放库存，可以通过领域事件最终一致
```

如果你试图让订单、库存、支付、优惠券全部在一个大事务里强一致，系统会越来越复杂，也越来越脆弱。

DDD 更鼓励你明确区分：

```text
哪些必须立刻一致
哪些可以最终一致
```

这也是聚合边界设计的核心问题。

## 二十三、什么时候不用完整 DDD？

DDD 不是银弹。

如果你的系统只是简单 CRUD：

```text
菜单管理
角色管理
文章分类
系统参数配置
简单审批流
```

不一定需要完整 DDD。

直接使用：

```text
Controller
Service
Repository
Model
```

可能更简单。

但如果你的系统有这些特征，就很适合引入 DDD：

```text
业务规则复杂
状态流转复杂
一个操作影响多个模块
业务概念多且容易混淆
业务变化频繁
Service 越写越大
大量 if else 到处重复
不同团队对同一个词理解不一致
```

DDD 的价值不是让代码更“高级”，而是让复杂业务有清晰的表达方式。

## 二十四、常见错误

### 错误一：把 DDD 做成目录游戏

有些项目只是把目录改成：

```text
domain
application
infrastructure
interfaces
```

但业务规则仍然全部写在 Service 里。

这不是真正的 DDD。

DDD 的关键不是目录，而是领域模型是否承载了业务规则。

### 错误二：一个实体对应一张表

DDD 里的实体不是数据库 Entity。

数据库 Entity 通常是 ORM 映射对象。DDD 里的实体是业务对象。

它们可以分开。

例如：

```text
Order 是领域聚合
OrderOrmEntity 是数据库映射对象
OrderRepository 负责两者转换
```

### 错误三：聚合设计过大

不要把客户、订单、支付、物流全部放进一个聚合。

聚合太大会导致：

```text
加载成本高
事务范围大
并发冲突多
模型难以维护
```

聚合应该尽量小，只保护真正需要强一致的规则。

### 错误四：领域对象依赖基础设施

领域对象里不应该出现：

```ts
import axios from 'axios'
import { prisma } from '@/db'
import { redis } from '@/redis'
```

如果领域对象依赖了 HTTP、ORM、Redis，它就很难测试，也很难复用。

### 错误五：所有跨模块操作都直接调用

如果订单取消后直接调用库存、支付、积分、通知，订单模块会越来越重。

更好的方式是：

```text
订单发布 OrderCancelled
库存、支付、积分、通知分别订阅事件
```

这样模块之间更松耦合。

## 二十五、一份实用的 DDD 设计检查清单

拿到一个需求后，可以按下面的问题逐个回答：

```text
1. 这个需求属于哪个业务上下文？
2. 这个需求的命令是什么？
3. 成功后会产生什么领域事件？
4. 过程中有哪些业务规则？
5. 哪些对象是实体？
6. 哪些对象是值对象？
7. 哪些对象必须强一致？
8. 谁是聚合根？
9. 聚合内部有哪些状态变化方法？
10. 哪些逻辑属于应用服务编排？
11. 是否需要调用其他上下文？
12. 如果需要，是否应该通过反腐层或领域事件隔离？
13. Repository 需要保存哪个聚合？
14. 数据库如何映射这个聚合？
15. 哪些地方需要最终一致？
```

如果这些问题回答清楚，模型基本就出来了。

## 二十六、最后总结

DDD 的核心不是“多建几个目录”，也不是“所有项目都要上复杂架构”。

它真正关注的是：

```text
业务概念是否清晰
业务边界是否明确
业务规则是否有归属
代码语言是否贴近业务语言
复杂流程是否可以被拆开维护
```

真实项目做 DDD，可以记住这条主线：

```text
先理解业务动作
再提炼业务规则
然后划分业务边界
接着设计聚合和值对象
最后再考虑数据库、接口和技术实现
```

最重要的一句话是：

> DDD 不是让简单系统变复杂，而是让复杂业务不失控。

当你发现一个 Service 里塞满状态判断、跨模块调用和重复规则时，就应该停下来问一句：

> 这些规则到底属于哪个业务概念？

找到这个答案，才是真正开始做 DDD 设计。
