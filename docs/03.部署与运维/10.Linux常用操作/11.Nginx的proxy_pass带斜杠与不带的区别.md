## Nginx的proxy_pass带斜杠与不带的区别

> 下文统一约定：后端服务地址为 `http://api-testcase:5001`，用户请求路径为 `/api/v1/resource`，重点观察转发到后端的最终路径。

## 一、proxy_pass 带不带尾斜杠的区别

判断 `proxy_pass` 如何改写路径，**关键不是“带不带斜杠”，而是 `proxy_pass` 的 URL 里有没有“路径（URI）”**：

- `http://api-testcase:5001` —— host:port 后面什么都没有，属于**无 URI**
- `http://api-testcase:5001/` —— host:port 后面有 `/`，属于**有 URI**（路径是根 `/`）
- `http://api-testcase:5001/service/` —— 属于**有 URI**（路径是 `/service/`）

### 1. 无 URI：原样传递完整请求路径

`proxy_pass` 后面只有 `host:port`、没有任何路径时，Nginx 会把**完整的原始请求 URI** 原封不动地交给后端，`location` 的匹配前缀不会被去掉。

```shell
location /api/ {
    proxy_pass http://api-testcase:5001;
}
```

- 请求：`/api/v1/resource`
- 转发到后端：`http://api-testcase:5001/api/v1/resource`

### 2. 有 URI：用 proxy_pass 的路径替换 location 匹配部分

`proxy_pass` 后面带了路径（哪怕只是根 `/`）时，Nginx 会把请求 URI 中**与 `location` 匹配的那段前缀**替换为 `proxy_pass` 里的路径，后面的部分保持不变。

```shell
location /api/ {
    proxy_pass http://api-testcase:5001/;
}
```

- 请求：`/api/v1/resource`
- `location` 匹配掉的段：`/api/`
- 替换为 `proxy_pass` 的路径：`/`
- 转发到后端：`http://api-testcase:5001/v1/resource`

如果把 `proxy_pass` 的路径换成 `/service/`，效果就是再前缀一段：

```shell
location /api/ {
    proxy_pass http://api-testcase:5001/service/;
}
```

- 请求：`/api/v1/resource`
- 转发到后端：`http://api-testcase:5001/service/v1/resource`

### 小结

| proxy_pass 形式 | 是否含 URI | 行为 | `/api/v1/resource` 转发结果 |
| -------------- | --------- | ---- | -------------------------- |
| `http://host`  | 无        | 原样传递完整请求 URI | `/api/v1/resource` |
| `http://host/` | 有（`/`） | 用 `/` 替换 location 匹配段 | `/v1/resource` |
| `http://host/service/` | 有（`/service/`） | 用 `/service/` 替换 location 匹配段 | `/service/v1/resource` |

## 二、location 带不带尾斜杠的区别

`location` 的尾斜杠决定的是**前缀匹配的范围**，和 `proxy_pass` 的改写规则无关。

### 1. location /api（不带斜杠）

属于普通前缀匹配，只要请求 URI **以 `/api` 开头**就会命中，包括 `/apixxx` 这种意料之外的路径：

```shell
location /api {
    ...
}
```

- `/api` —— 命中
- `/api/v1/resource` —— 命中
- `/apixxx` —— **也命中**（前缀就是 `/api`）

### 2. location /api/（带斜杠）

同样是前缀匹配，但前缀变成了 `/api/`，因此只匹配以 `/api/` 开头的请求，能避开上面的误命中：

```shell
location /api/ {
    ...
}
```

- `/api/` —— 命中
- `/api/v1/resource` —— 命中
- `/apixxx` —— **不命中**
- `/api`（无尾斜杠）—— **不命中**

### 小结

| location 写法 | 匹配前缀 | `/api` | `/api/` | `/api/x` | `/apixxx` |
| ------------- | -------- | ------ | ------- | -------- | --------- |
| `/api`        | `/api`   | ✓      | ✓       | ✓        | ✓         |
| `/api/`       | `/api/`  | ✗      | ✓       | ✓        | ✗         |

> 建议：对外提供接口时优先用 `location /api/`，避免 `/apixxx` 之类的路径被意外匹配到。

## 三、四种组合速查

把上面两个维度组合起来，就是日常最容易混淆的四种写法：

| location | proxy_pass | 请求路径 | 实际转发路径 | 结果 |
| -------- | ---------- | -------- | ------------ | ---- |
| `/api`   | `.../`     | `/api/x` | `/x`         | 去掉 `/api` |
| `/api`   | `...`      | `/api/x` | `/api/x`     | 保留 `/api` |
| `/api/`  | `.../`     | `/api/x` | `/x`         | 去掉 `/api/` |
| `/api/`  | `...`      | `/api/x` | `/api/x`     | 保留 `/api/` |

> 口诀：`proxy_pass` 带 URI 就替换、不带 URI 就原样传；`location` 带斜杠就收紧匹配范围。
