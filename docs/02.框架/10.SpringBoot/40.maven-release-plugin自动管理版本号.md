# Maven Release Plugin 实战：让 Maven 自动管理版本、Git Tag 与制品发布

在 Maven 项目里，发布一个正式版本看起来只是把 `1.2.3-SNAPSHOT` 改成 `1.2.3`。真正做起来，却往往还有一串容易遗漏的动作：提交版本修改、创建 Git Tag、上传制品、再把主分支切换到下一个开发版本。

`maven-release-plugin` 就是用来自动完成这套流程的。

本文从一个实际的 Git + Maven + Nexus 项目出发，完整讲清楚插件如何工作、怎样配置、如何在本地和 Jenkins 中执行，以及发布失败后应该怎样恢复。

> 本文示例使用 `maven-release-plugin 3.3.1`。如果你的公司统一管理 Maven 插件版本，应以公司父 POM 中的版本为准。

## 一、它到底自动管理了什么

假设项目目前处于开发阶段，`pom.xml` 中的版本是：

```xml
<version>1.2.3-SNAPSHOT</version>
```

执行一次完整发布后，仓库中会发生下面这些变化：

```text
开始状态
└── 主分支版本：1.2.3-SNAPSHOT

准备发布
├── 修改版本：1.2.3-SNAPSHOT -> 1.2.3
├── 执行测试
├── 提交 release 版本
└── 创建 Git Tag：v1.2.3

继续开发
├── 修改版本：1.2.3 -> 1.2.4-SNAPSHOT
└── 提交下一个开发版本

正式部署
└── 从 v1.2.3 重新检出代码并执行 mvn deploy
```

最后得到三个清晰的结果：

| 位置 | 版本 | 用途 |
| --- | --- | --- |
| Git Tag `v1.2.3` | `1.2.3` | 对应已经发布的不可变源码 |
| Maven Release 仓库 | `1.2.3` | 供其他项目正式依赖 |
| Git 主分支 | `1.2.4-SNAPSHOT` | 继续下一轮开发 |

这里最容易忽略的一点是：**Release Plugin 不只是改版本号，它还会直接操作 Git，并且会提交和推送代码。** 因此，运行发布命令的账号必须同时具备代码仓库和制品仓库的写权限。

## 二、`prepare` 和 `perform` 为什么要分成两步

Maven Release Plugin 把发布拆成两个阶段。

### 1. `release:prepare`

`prepare` 负责准备源码版本。默认流程包括：

1. 检查工作区是否存在未提交修改。
2. 检查项目是否依赖不允许发布的 `SNAPSHOT` 版本。
3. 把当前版本改成正式版本。
4. 执行构建和测试。
5. 提交正式版本。
6. 创建 Git Tag。
7. 把主分支改成下一个 `SNAPSHOT` 版本。
8. 再提交一次版本修改。

这一步完成后，正式发布版本已经被 Git Tag 固定下来。

### 2. `release:perform`

`perform` 不直接使用当前工作区。它会根据 `release:prepare` 生成的 `release.properties`，重新检出刚刚创建的 Tag，然后执行发布目标，默认通常是：

```bash
mvn deploy
```

默认检出目录是：

```text
target/checkout
```

这样做的好处很实际：上传到 Nexus 或 Artifactory 的构件，确实来自 Git Tag 中的源码，而不是来自开发者电脑上某个未经记录的临时状态。

## 三、发布前需要满足的条件

接入插件之前，先确认下面几件事。

### Maven 项目版本必须是 `SNAPSHOT`

日常开发版本应类似：

```xml
<version>1.2.3-SNAPSHOT</version>
```

如果项目当前已经是 `1.2.3`，插件无法按照正常的开发版本流程计算下一步。

### 项目必须位于版本控制仓库中

Release Plugin 需要从 `pom.xml` 的 `<scm>` 节点获取仓库地址，并执行提交、打 Tag 和检出操作。

### 当前工作区必须干净

发布前执行：

```bash
git status
```

理想输出是：

```text
nothing to commit, working tree clean
```

未跟踪文件、被修改但未提交的配置文件，都可能让 `release:prepare` 中止。不要为了绕过检查而关闭安全选项，先弄清这些修改是否应该进入发布版本。

### 发布账号需要两类权限

- Git 权限：提交代码、推送提交、创建并推送 Tag。
- Maven 仓库权限：向 Release 仓库上传正式构件。

如果主分支受保护，不允许 CI 账号直接推送，那么标准 Release Plugin 流程会被阻止。此时需要为发布账号配置权限，或者改用不直接提交版本号的 CI Friendly Versions 等流程。

## 四、完整的 `pom.xml` 配置

下面是一份可用于 Git + Nexus 的单模块示例。请把域名、组织名和仓库路径换成自己的实际值。

```xml
<?xml version="1.0" encoding="UTF-8"?>
<project xmlns="http://maven.apache.org/POM/4.0.0"
         xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
         xsi:schemaLocation="http://maven.apache.org/POM/4.0.0
                             https://maven.apache.org/xsd/maven-4.0.0.xsd">
    <modelVersion>4.0.0</modelVersion>

    <groupId>com.example</groupId>
    <artifactId>order-service</artifactId>
    <version>1.2.3-SNAPSHOT</version>
    <packaging>jar</packaging>

    <properties>
        <maven.compiler.release>17</maven.compiler.release>
        <project.build.sourceEncoding>UTF-8</project.build.sourceEncoding>
        <maven-release-plugin.version>3.3.1</maven-release-plugin.version>
    </properties>

    <!-- Release Plugin 通过这里定位 Git 仓库 -->
    <scm>
        <!-- 只读连接，可使用 HTTPS -->
        <connection>
            scm:git:https://git.example.com/backend/order-service.git
        </connection>

        <!-- 需要提交和创建 Tag，因此通常使用有写权限的 SSH 地址 -->
        <developerConnection>
            scm:git:ssh://git@git.example.com/backend/order-service.git
        </developerConnection>

        <!-- 供人浏览的仓库页面 -->
        <url>https://git.example.com/backend/order-service</url>
        <tag>HEAD</tag>
    </scm>

    <!-- deploy 阶段向这里上传构件 -->
    <distributionManagement>
        <repository>
            <id>company-releases</id>
            <name>Company Release Repository</name>
            <url>https://nexus.example.com/repository/maven-releases/</url>
        </repository>

        <snapshotRepository>
            <id>company-snapshots</id>
            <name>Company Snapshot Repository</name>
            <url>https://nexus.example.com/repository/maven-snapshots/</url>
        </snapshotRepository>
    </distributionManagement>

    <build>
        <plugins>
            <plugin>
                <groupId>org.apache.maven.plugins</groupId>
                <artifactId>maven-release-plugin</artifactId>
                <version>${maven-release-plugin.version}</version>
                <configuration>
                    <!-- Tag 最终形如 v1.2.3 -->
                    <tagNameFormat>v@{project.version}</tagNameFormat>

                    <!-- prepare 阶段用于验证发布版本 -->
                    <preparationGoals>clean verify</preparationGoals>

                    <!-- perform 阶段执行的目标 -->
                    <goals>deploy</goals>

                    <!-- 允许插件推送版本提交和 Tag -->
                    <pushChanges>true</pushChanges>
                </configuration>
            </plugin>
        </plugins>
    </build>
</project>
```

### 为什么 Tag 格式写成 `v@{project.version}`

这里必须使用：

```xml
<tagNameFormat>v@{project.version}</tagNameFormat>
```

不要写成：

```xml
<tagNameFormat>v${project.version}</tagNameFormat>
```

`${project.version}` 会在 Maven 正常解析 POM 时提前展开，很可能得到开发版本；`@{project.version}` 会等到 Release Plugin 完成版本转换后再取值。

## 五、配置 Nexus 或 Artifactory 账号

`pom.xml` 可以保存仓库地址，但不应该保存用户名和密码。部署凭据应放在 Maven 的 `settings.xml` 中。

默认文件位置是：

```text
~/.m2/settings.xml
```

示例：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<settings xmlns="http://maven.apache.org/SETTINGS/1.2.0"
          xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
          xsi:schemaLocation="http://maven.apache.org/SETTINGS/1.2.0
                              https://maven.apache.org/xsd/settings-1.2.0.xsd">
    <servers>
        <server>
            <id>company-releases</id>
            <username>${env.NEXUS_USERNAME}</username>
            <password>${env.NEXUS_PASSWORD}</password>
        </server>

        <server>
            <id>company-snapshots</id>
            <username>${env.NEXUS_USERNAME}</username>
            <password>${env.NEXUS_PASSWORD}</password>
        </server>
    </servers>
</settings>
```

两个地方的 `id` 必须一致：

```text
pom.xml                           settings.xml
distributionManagement/id   <->  servers/server/id
company-releases             <->  company-releases
```

本地可以通过环境变量提供凭据：

```bash
export NEXUS_USERNAME='release-user'
export NEXUS_PASSWORD='your-secret'
```

在 Jenkins、GitLab CI 或 GitHub Actions 中，应使用平台的凭据管理功能注入变量或生成临时 `settings.xml`，不要把真实密码提交到代码仓库。

Git 认证与 Nexus 认证是两套独立凭据。上面的 `settings.xml` 只解决制品上传问题；Git SSH 地址仍需要本机或 CI 环境能够访问对应私钥。

## 六、第一次发布：建议分阶段执行

假设当前版本是：

```xml
<version>1.2.3-SNAPSHOT</version>
```

### 第一步：发布前验证

```bash
mvn clean verify
git status
git branch --show-current
```

至少确认：

- 测试能够通过。
- 工作区没有未提交修改。
- 当前分支就是准备发布的分支。
- 当前提交已经同步到远程仓库。
- Git 用户具有推送提交和 Tag 的权限。

CI 环境还应配置明确的提交身份：

```bash
git config user.name "Maven Release Bot"
git config user.email "release-bot@example.com"
```

### 第二步：先做一次 Dry Run

第一次接入时，可以先模拟 `prepare`：

```bash
mvn release:clean release:prepare -DdryRun=true
```

Dry Run 会执行大部分检查和版本计算，但不会真正提交或创建 Tag。检查无误后清理模拟文件：

```bash
mvn release:clean
```

Dry Run 不能证明 Git 推送权限和 Nexus 上传权限一定正常，它只适合提前发现 POM、版本和构建方面的问题。

### 第三步：准备正式版本

```bash
mvn release:clean release:prepare
```

交互过程大致如下：

```text
What is the release version for "order-service"? (1.2.3) 1.2.3
What is SCM release tag or label for "order-service"? (v1.2.3) v1.2.3
What is the new development version for "order-service"? (1.2.4-SNAPSHOT) 1.2.4-SNAPSHOT
```

如果接受括号中的默认值，直接按回车即可。

`prepare` 成功后，可以检查 Git 状态：

```bash
git log --oneline --decorate -5
git tag --list 'v1.2.3'
git status
```

典型提交历史类似：

```text
8b41d62 (HEAD -> main) [maven-release-plugin] prepare for next development iteration
5ea31bd (tag: v1.2.3) [maven-release-plugin] prepare release v1.2.3
09a8e21 feat: complete order cancellation
```

这时主分支的 `pom.xml` 已经变成：

```xml
<version>1.2.4-SNAPSHOT</version>
```

而 `v1.2.3` Tag 中保存的是：

```xml
<version>1.2.3</version>
```

### 第四步：部署正式构件

```bash
mvn release:perform
```

插件会从 `v1.2.3` 检出一个干净副本，并在其中执行：

```bash
mvn deploy
```

发布成功后，可以在 Nexus 中看到类似路径：

```text
com/example/order-service/1.2.3/
├── order-service-1.2.3.jar
├── order-service-1.2.3.pom
├── order-service-1.2.3.jar.sha1
└── order-service-1.2.3.pom.sha1
```

其他项目随后可以依赖这个正式版本：

```xml
<dependency>
    <groupId>com.example</groupId>
    <artifactId>order-service</artifactId>
    <version>1.2.3</version>
</dependency>
```

确认流程稳定后，也可以一次执行完：

```bash
mvn release:clean release:prepare release:perform
```

生产环境第一次接入时，仍然建议把 `prepare` 和 `perform` 分开。这样如果上传仓库失败，可以明确知道 Git 版本已经准备完成，只需修复部署问题后重试 `perform`。

## 七、非交互式发布：适合 CI

CI 中不应该等待人工回答版本号。最稳妥的方式是显式传入正式版本、下一个开发版本和 Tag：

```bash
mvn --batch-mode \
  release:clean \
  release:prepare \
  release:perform \
  -DreleaseVersion=1.4.0 \
  -DdevelopmentVersion=1.4.1-SNAPSHOT \
  -Dtag=v1.4.0
```

其中：

| 参数 | 示例 | 含义 |
| --- | --- | --- |
| `releaseVersion` | `1.4.0` | 本次发布的正式版本 |
| `developmentVersion` | `1.4.1-SNAPSHOT` | 发布后主分支的新版本 |
| `tag` | `v1.4.0` | 本次发布创建的 Git Tag |
| `--batch-mode` / `-B` | 无值 | 禁止交互，适合 CI 日志 |

如果发布构建需要额外 Profile，可以在 POM 中配置：

```xml
<configuration>
    <releaseProfiles>release</releaseProfiles>
</configuration>
```

或者通过 `arguments` 把参数传给插件内部启动的 Maven：

```bash
mvn -B release:clean release:prepare release:perform \
  -DreleaseVersion=1.4.0 \
  -DdevelopmentVersion=1.4.1-SNAPSHOT \
  -Dtag=v1.4.0 \
  -Darguments="-DskipTests=false -Pproduction"
```

之所以需要 `-Darguments`，是因为 `release:prepare` 和 `release:perform` 会启动新的 Maven 进程。只写在最外层命令上的某些构建参数，不一定会自动传入内部构建。

## 八、Jenkins Pipeline 示例

下面的示例假设 Jenkins 中已经配置：

- SSH 私钥凭据：`git-release-key`
- Maven `settings.xml` 文件凭据：`maven-settings`
- Jenkins 节点已经安装 Java、Maven 和 Git
- 使用 SSH Agent Plugin 提供 `sshagent` 步骤

```groovy
pipeline {
    agent any

    options {
        disableConcurrentBuilds()
        timestamps()
    }

    parameters {
        string(
            name: 'RELEASE_VERSION',
            defaultValue: '1.4.0',
            description: '本次正式发布版本，例如 1.4.0'
        )
        string(
            name: 'NEXT_VERSION',
            defaultValue: '1.4.1-SNAPSHOT',
            description: '发布后的开发版本，例如 1.4.1-SNAPSHOT'
        )
    }

    stages {
        stage('Checkout') {
            steps {
                // 发布任务应完整检出，不要使用只包含一个提交的浅克隆。
                checkout scm
            }
        }

        stage('Verify') {
            steps {
                sh 'mvn -B clean verify'
                sh 'git status --short'
            }
        }

        stage('Release') {
            steps {
                withCredentials([
                    file(
                        credentialsId: 'maven-settings',
                        variable: 'MAVEN_SETTINGS'
                    )
                ]) {
                    sshagent(credentials: ['git-release-key']) {
                        sh '''
                            git config user.name "Maven Release Bot"
                            git config user.email "release-bot@example.com"

                            mvn -B -s "$MAVEN_SETTINGS" \
                              release:clean \
                              release:prepare \
                              release:perform \
                              -DreleaseVersion="$RELEASE_VERSION" \
                              -DdevelopmentVersion="$NEXT_VERSION" \
                              -Dtag="v$RELEASE_VERSION"
                        '''
                    }
                }
            }
        }
    }
}
```

这段 Pipeline 还需要配合实际仓库策略调整。尤其要注意：

- 不要让同一项目同时运行两个发布任务，否则两个任务可能争用同一个版本。
- 不要从 Pull Request 的临时合并提交执行正式发布。
- Jenkins 默认检出可能处于 detached HEAD，需确认所用 SCM 配置允许 Release Plugin 推送到正确分支。
- 如果主分支禁止直接推送，先为发布机器人设计明确的授权方案。
- 版本参数应在任务开始前校验，避免把 `1.4`、`v1.4.0` 或带空格的值传进错误字段。

## 九、多模块项目怎么配置

假设项目结构是：

```text
shop-platform/
├── pom.xml
├── shop-api/
│   └── pom.xml
├── shop-service/
│   └── pom.xml
└── shop-web/
    └── pom.xml
```

父 POM：

```xml
<groupId>com.example.shop</groupId>
<artifactId>shop-platform</artifactId>
<version>2.3.0-SNAPSHOT</version>
<packaging>pom</packaging>

<modules>
    <module>shop-api</module>
    <module>shop-service</module>
    <module>shop-web</module>
</modules>
```

子模块继承父版本：

```xml
<parent>
    <groupId>com.example.shop</groupId>
    <artifactId>shop-platform</artifactId>
    <version>2.3.0-SNAPSHOT</version>
</parent>

<artifactId>shop-service</artifactId>
```

如果所有模块跟随父项目统一发布，在父 POM 中加入：

```xml
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-release-plugin</artifactId>
    <version>3.3.1</version>
    <configuration>
        <autoVersionSubmodules>true</autoVersionSubmodules>
        <tagNameFormat>v@{project.version}</tagNameFormat>
        <preparationGoals>clean verify</preparationGoals>
        <goals>deploy</goals>
    </configuration>
</plugin>
```

然后执行：

```bash
mvn -B release:clean release:prepare release:perform \
  -DreleaseVersion=2.3.0 \
  -DdevelopmentVersion=2.3.1-SNAPSHOT \
  -Dtag=v2.3.0 \
  -DautoVersionSubmodules=true
```

这样父项目和所有子模块会一起从 `2.3.0-SNAPSHOT` 切换到 `2.3.0`，发布后再统一切到 `2.3.1-SNAPSHOT`。

如果各模块独立发布、版本互不相同，就不应该启用 `autoVersionSubmodules`。Release Plugin 会分别处理模块版本，但这种仓库结构的发布和回滚成本明显更高。实践中，真正需要独立生命周期的模块通常更适合拆分仓库或建立独立发布流水线。

## 十、如何控制下一个版本

默认版本策略适合最常见的补丁版本递增：

```text
当前版本：1.2.3-SNAPSHOT
正式版本：1.2.3
开发版本：1.2.4-SNAPSHOT
```

新版 Release Plugin 还提供 SemVer 版本策略。比如希望发布后进入下一个次版本开发，可以配置：

```xml
<configuration>
    <projectVersionPolicyId>
        SemVerMinorDevelopment
    </projectVersionPolicyId>
</configuration>
```

效果是：

```text
当前版本：1.2.3-SNAPSHOT
正式版本：1.2.3
开发版本：1.3.0-SNAPSHOT
```

常见策略如下：

| 策略 | 当前版本 | 正式版本 | 下一个开发版本 |
| --- | --- | --- | --- |
| `default` | `1.2.3-SNAPSHOT` | `1.2.3` | `1.2.4-SNAPSHOT` |
| `SemVerPatchDevelopment` | `1.2.3-SNAPSHOT` | `1.2.3` | `1.2.4-SNAPSHOT` |
| `SemVerMinorDevelopment` | `1.2.3-SNAPSHOT` | `1.2.3` | `1.3.0-SNAPSHOT` |
| `SemVerMajorDevelopment` | `1.2.3-SNAPSHOT` | `1.2.3` | `2.0.0-SNAPSHOT` |

对于生产发布流水线，我更建议直接传入：

```bash
-DreleaseVersion=1.2.3
-DdevelopmentVersion=1.3.0-SNAPSHOT
```

版本策略适合提供默认值，显式参数则更容易审计，也能避免团队对“下一版本应该升 major、minor 还是 patch”产生不同理解。

## 十一、失败以后怎么恢复

发布失败时，先判断失败发生在哪个阶段，不要立刻删除 Tag 或手工改 POM。

### 情况一：`prepare` 在提交前失败

常见原因是测试失败、存在 `SNAPSHOT` 依赖或工作区不干净。

修复问题后，可以直接继续：

```bash
mvn release:prepare
```

Release Plugin 默认会根据 `release.properties` 尝试从中断位置恢复。

如果希望放弃本次状态并重新开始，先检查 POM 是否仍有版本修改：

```bash
git status --short
```

如果工作区已经恢复干净，可以清理状态并重新执行：

```bash
mvn release:clean release:prepare
```

如果 POM 仍然被上一次 `prepare` 修改，不要先运行 `release:clean`，因为它会删除回滚需要的备份文件。此时应先使用下一节的 `release:rollback`，确认工作区恢复后再重新发布。

### 情况二：`prepare` 已修改 POM，但希望撤销

在 `release.properties` 仍然存在时，可以执行：

```bash
mvn release:rollback
```

它用于回滚上一次 `prepare` 产生的版本变更。执行前仍应检查远程仓库中是否已经存在提交或 Tag，以及团队是否允许改写或删除这些发布记录。

### 情况三：`prepare` 成功，`perform` 上传失败

此时通常不需要重新打 Tag。只要当前工作区中的 `release.properties` 仍然存在，修复 Nexus 凭据、网络或签名问题后即可重新执行：

```bash
mvn release:perform
```

它会继续使用 `release.properties` 中记录的 Tag。如果这个文件已经被清理，不要直接重新运行完整的 `prepare`；先根据远程提交、Tag 和制品仓库的实际状态决定恢复方式，避免重复创建发布记录。

### 情况四：只想清理本地残留文件

```bash
mvn release:clean
```

这个命令会清理 `release.properties` 和备份 POM，但**不会撤销已经推送到远程 Git 仓库的提交或 Tag**。

### 情况五：正式构件已经上传，但发布内容有问题

大多数 Maven Release 仓库不允许覆盖同一个正式版本，这是正确的仓库策略。不要试图重新上传一个内容不同的 `1.2.3`。

更稳妥的处理方式是：

1. 保留 `1.2.3` 的 Git Tag 和制品记录。
2. 修复问题。
3. 发布新的 `1.2.4`。

已经被其他项目下载过的正式版本应视为不可变数据。

## 十二、最常见的报错与排查方法

### `You don't have a SNAPSHOT project in the reactor projects list`

项目当前版本不是 `SNAPSHOT`。检查父 POM、子模块及命令实际加载的 Profile：

```bash
mvn help:evaluate -Dexpression=project.version -q -DforceStdout
```

### `Cannot prepare the release because you have local modifications`

工作区存在修改。执行：

```bash
git status --short
```

判断这些文件应该提交、忽略还是恢复，不要直接关闭检查。

### `Can't release project due to non released dependencies`

项目仍然依赖其他 `SNAPSHOT` 构件，例如：

```xml
<dependency>
    <groupId>com.example</groupId>
    <artifactId>common-core</artifactId>
    <version>2.0.0-SNAPSHOT</version>
</dependency>
```

应先发布 `common-core` 的正式版本，再把依赖改成正式版本。强行允许 `SNAPSHOT` 会让正式构件失去可重复构建能力。

### Git 推送被拒绝

重点检查：

```bash
git remote -v
ssh -T git@git.example.com
git branch --show-current
```

还需要查看分支保护、Tag 保护、CI 账号权限，以及当前是否处于 detached HEAD。

### `401 Unauthorized` 或 `403 Forbidden`

这通常是 Maven 仓库认证问题。检查：

- `distributionManagement.repository.id`
- `settings.xml` 中的 `server.id`
- CI 是否正确注入用户名和密码
- 账号是否拥有 Release 仓库部署权限
- 仓库 URL 是否指向可写的 hosted repository，而不是只读 group repository

可以先用一个测试版本验证普通部署链路：

```bash
mvn -s /path/to/settings.xml deploy
```

不要用已经发布过的正式版本反复测试，因为 Release 仓库通常禁止覆盖。

### `perform` 找不到 Tag 或检出失败

常见原因包括：

- CI 使用浅克隆。
- `developerConnection` 地址写错。
- Tag 没有成功推送到远程。
- `release.properties` 来自另一个工作区或旧任务。
- Git 凭据只能推送，不能从目标地址重新检出。

先确认远程 Tag：

```bash
git ls-remote --tags origin v1.2.3
```
