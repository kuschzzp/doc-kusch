# Spring Cloud 核心组件

官方文档：[Spring Cloud Reference](https://docs.spring.io/spring-cloud/docs/current/reference/html/)

Spring Cloud 不是一个单独的框架，而是一组微服务常见问题的解决方案集合。

它主要解决这些问题：

- 配置怎么集中管理。
- 服务之间怎么发现彼此。
- 服务调用怎么负载均衡。
- 网关怎么统一入口。
- 服务失败后怎么熔断、降级。
- 消息怎么在系统间流动。
- 调用链怎么观测。

## 现在看 Spring Cloud 要注意版本变化

很多老文章还在讲：

```text
Eureka + Ribbon + Hystrix + Zuul + Sleuth
```

这套组合现在已经不是主流推荐了。

更应该关注：

```text
Nacos / Consul / Eureka       服务注册与发现
Spring Cloud Config / Nacos   配置中心
Spring Cloud LoadBalancer     客户端负载均衡
Spring Cloud Gateway          网关
Spring Cloud OpenFeign        声明式 HTTP 调用
Spring Cloud CircuitBreaker   熔断抽象，常配 Resilience4j
Micrometer Tracing            链路追踪方向，Sleuth 已进入维护/迁移阶段
Spring Cloud Stream           消息驱动
```

## 一张简单架构图

```text
                 Config / Nacos
                      │
                      ▼
            Service Discovery
             Nacos / Eureka
                      │
                      ▼
Client -> Spring Cloud Gateway
                      │
          ┌───────────┴───────────┐
          ▼                       ▼
      user-service            order-service
          │                       │
          └──── OpenFeign / LoadBalancer ────► other-service
```

## 服务注册与发现

服务注册中心解决的是：服务实例地址会变，调用方不能写死 IP。

常见选择：

- Nacos
- Eureka
- Consul
- Zookeeper

如果是国内项目，Nacos 比较常见，因为它同时提供服务发现和配置管理。

如果是纯 Spring 官方生态，也可以看 Eureka、Consul、Zookeeper。

## 配置中心

配置中心解决的是：配置不要散落在每个服务的 jar 包里。

常见选择：

- Spring Cloud Config
- Nacos Config
- Apollo

Spring Cloud Config 默认很适合 Git 管理配置。

Nacos Config 更偏“一站式”：服务注册、配置管理都放在 Nacos 里。

## 服务调用

服务之间调用常见两种方式：

- `RestTemplate` / `RestClient` / `WebClient` + `@LoadBalanced`
- OpenFeign

现在更常用 OpenFeign，因为接口声明更清楚。

```java
@FeignClient(name = "user-service")
public interface UserClient {
    @GetMapping("/users/{id}")
    UserDTO getById(@PathVariable("id") Long id);
}
```

## 负载均衡

Ribbon 已经不是新项目推荐方向。

现在 Spring Cloud 推荐的是 Spring Cloud LoadBalancer。

它是客户端负载均衡：调用方根据服务名拿到多个实例，再选一个实例发请求。

默认常见策略是轮询，也可以扩展成随机、权重、区域优先等。

## 网关

Spring Cloud Gateway 是现在 Spring Cloud 里常用的网关。

它基于 Spring WebFlux / Reactor Netty，不适合打成传统 Servlet WAR 部署。

它的三个核心概念：

```text
Route      路由规则
Predicate 断言，判断请求是否匹配
Filter    过滤器，修改请求或响应
```

## 熔断与降级

Hystrix 已经不是新项目推荐方向。

现在更多使用 Spring Cloud CircuitBreaker 这一层抽象，底层常配 Resilience4j。

它主要处理：

- 远程调用失败。
- 调用超时。
- 下游服务异常。
- 防止故障扩散。

## 消息与事件

Spring Cloud Stream 用来做消息驱动应用。

它把应用和具体消息中间件隔开，底层可以接 Kafka、RabbitMQ 等 binder。

Spring Cloud Bus 则常用于广播事件，比如配置刷新。

## 链路追踪

老项目里常见 Spring Cloud Sleuth + Zipkin。

现在新项目更应该关注 Micrometer Tracing 和 OpenTelemetry 方向。

链路追踪解决的是：一个请求经过多个服务后，怎么知道慢在哪里、错在哪里。

## 小结

Spring Cloud 可以按问题域记：

```text
配置中心        Config / Nacos Config
注册发现        Nacos / Eureka / Consul
服务调用        OpenFeign / RestClient / WebClient
负载均衡        Spring Cloud LoadBalancer
网关            Spring Cloud Gateway
熔断降级        Spring Cloud CircuitBreaker + Resilience4j
消息            Spring Cloud Stream / Bus
链路追踪        Micrometer Tracing / OpenTelemetry
```

学习时不要只背组件名，要先知道它解决的是哪个分布式系统问题。
