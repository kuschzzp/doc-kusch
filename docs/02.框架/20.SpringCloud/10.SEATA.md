# Seata 分布式事务

官方文档：[Apache Seata](https://seata.apache.org/docs/overview/what-is-seata/)

Seata 解决的是分布式事务问题。

比如一个下单流程：

```text
order-service  创建订单
account-service 扣余额
stock-service   扣库存
```

这三个服务都有自己的数据库。

如果订单创建成功、余额扣了，但库存扣失败，就会出现数据不一致。

Seata 就是为了解决这类跨服务、跨数据库的一致性问题。

## Seata 的几个角色

官方里常见三个角色：

```text
TC  Transaction Coordinator，全局事务协调者
TM  Transaction Manager，事务发起方
RM  Resource Manager，资源管理器
```

简单理解：

- TM：告诉 Seata 我要开启一个全局事务。
- RM：管理具体数据库资源。
- TC：协调各个分支事务提交或回滚。

## 常见事务模式

Seata 官方主要提供几种事务模式：

```text
AT   自动补偿型，业务侵入低
TCC  业务显式实现 Try / Confirm / Cancel
Saga 长事务补偿型，适合流程长、参与方多的业务
XA   基于数据库 XA 协议的强一致事务
```

它们不是谁替代谁，而是面向不同场景。

| 模式 | 一致性倾向 | 业务侵入 | 性能 | 典型场景 |
| --- | --- | --- | --- | --- |
| AT | 最终一致，自动补偿 | 低 | 较好 | 普通关系型数据库写操作，业务不想改太多代码 |
| TCC | 最终一致，业务补偿 | 高 | 好 | 核心交易、库存冻结、资金冻结等需要精确控制资源的场景 |
| Saga | 最终一致，正向/反向流程补偿 | 中到高 | 适合长流程 | 订单、审批、跨系统长事务，允许中间状态存在 |
| XA | 强一致，两阶段提交 | 低到中 | 较低 | 数据库支持 XA，且强一致优先于性能的场景 |

### AT 模式

AT 是 Spring Cloud Alibaba 项目里最常见的模式。

它的特点是业务侵入低。大多数时候你还是写普通的本地事务 SQL，Seata 通过数据源代理拦截 SQL，并自动记录回滚日志。

以扣库存为例：

```text
执行业务 SQL 前 -> 记录 before image
执行业务 SQL 后 -> 记录 after image
全局提交 -> 删除 undo_log
全局回滚 -> 根据 undo_log 生成反向补偿
```

AT 模式的关键依赖是 `undo_log` 表。

它适合：

- 关系型数据库。
- 普通增删改业务。
- 希望少改业务代码。
- 可以接受 Seata 代理数据源和 undo_log 机制。

不适合：

- 复杂 SQL 无法正确生成回滚镜像。
- 非关系型资源。
- 对全局锁冲突特别敏感的高并发热点更新。

### TCC 模式

TCC 是 Try、Confirm、Cancel 三个阶段。

```text
Try      预留资源
Confirm  确认提交
Cancel   取消预留
```

比如账户扣款，不是在 Try 阶段直接把钱扣掉，而是先冻结金额：

```text
Try      冻结 100 元
Confirm  真正扣减冻结金额
Cancel   解冻 100 元
```

TCC 的优点是业务控制力强，不依赖 Seata 自动生成反向 SQL。

但代价也明显：业务方必须自己实现三个方法，还要处理幂等、空回滚、悬挂等问题。

TCC 适合：

- 资金类业务。
- 库存冻结。
- 优惠券锁定。
- 需要明确资源预留和确认的核心链路。

TCC 不适合随手上。它对业务建模要求更高。

### Saga 模式

Saga 更适合长事务。

它把一个大事务拆成多个本地事务，每一步都有对应的补偿动作。

```text
创建订单        -> 取消订单
扣减库存        -> 恢复库存
扣减账户余额    -> 退回账户余额
生成物流单      -> 取消物流单
```

如果执行到中间失败，就按已经完成的步骤反向补偿。

Saga 的特点是：

- 不要求长时间锁资源。
- 适合跨服务、跨系统、流程较长的业务。
- 中间状态可能对外可见。
- 最终通过补偿达到一致。

它适合：

- 订单流程。
- 审批流程。
- 跨外部系统调用。
- 无法用本地数据库事务包住的长业务链路。

使用 Saga 要接受一个事实：它不是强一致，而是最终一致。

### XA 模式

XA 是数据库层面的分布式事务协议，典型是两阶段提交。

```text
第一阶段 prepare
第二阶段 commit / rollback
```

它的优点是强一致语义更明确，业务侵入相对小。

但问题也明显：

- 依赖数据库和驱动支持 XA。
- 资源锁持有时间更长。
- 性能和可用性压力更大。
- 参与方越多，整体复杂度越高。

XA 适合强一致优先的场景。

如果业务能接受最终一致，一般不会优先选 XA。

### 怎么选？

我的理解可以简单点：

```text
普通数据库写操作，想少改代码：先看 AT
核心资源预留，业务能明确 Try/Confirm/Cancel：看 TCC
长流程、跨系统、允许最终一致：看 Saga
强一致优先，数据库支持且能接受性能代价：看 XA
```

如果只是普通业务系统，最常先接触到的还是 AT。

但不要因为 AT 接入简单，就把所有跨服务流程都强行放进全局事务里。

## undo_log 表

AT 模式需要每个业务库都有 `undo_log` 表。

示意：

```sql
CREATE TABLE undo_log (
  branch_id BIGINT NOT NULL COMMENT 'branch transaction id',
  xid VARCHAR(128) NOT NULL COMMENT 'global transaction id',
  context VARCHAR(128) NOT NULL,
  rollback_info LONGBLOB NOT NULL,
  log_status INT NOT NULL,
  log_created DATETIME NOT NULL,
  log_modified DATETIME NOT NULL,
  UNIQUE KEY ux_undo_log (xid, branch_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

具体 SQL 以你使用的 Seata 版本官方脚本为准，不要从旧博客里复制一份就直接上生产。

## 使用 @GlobalTransactional

在全局事务入口方法上加：

```java
@GlobalTransactional(name = "create-order", rollbackFor = Exception.class)
public void createOrder(CreateOrderCommand command) {
    orderService.create(command);
    accountClient.debit(command.getUserId(), command.getAmount());
    stockClient.deduct(command.getProductId(), command.getCount());
}
```

这个方法通常在事务发起方，也就是 TM 所在服务。

## Spring Cloud Alibaba 集成思路

常见依赖：

```xml
<dependency>
  <groupId>com.alibaba.cloud</groupId>
  <artifactId>spring-cloud-starter-alibaba-seata</artifactId>
</dependency>
```

配置里通常要关心：

- 应用名。
- Seata 事务组。
- TC 地址或注册中心配置。
- 数据源代理。
- undo_log 表。

不同 Seata / Spring Cloud Alibaba 版本配置差异比较明显，实际项目一定要对照对应版本官方文档。

## 什么时候不要用 Seata？

不是所有跨服务流程都应该上分布式事务。

如果业务允许最终一致，可以优先考虑：

- 本地事务 + 消息。
- 事务消息。
- Outbox 模式。
- 补偿任务。
- Saga。

强一致分布式事务会增加复杂度，也会影响性能。

如果只是为了“写起来像单体事务”，直接上 Seata，后面排查问题会比较痛苦。

## 常见坑

### 1. 忘了 undo_log

AT 模式没有 `undo_log`，回滚就没有基础。

### 2. 事务边界太大

不要把一堆慢调用、外部接口、长时间逻辑都塞进全局事务。

### 3. 本地事务和全局事务边界混乱

`@Transactional` 和 `@GlobalTransactional` 要明确谁是入口，哪些方法只是分支。

### 4. 版本不匹配

Seata、Spring Cloud Alibaba、Spring Boot、Spring Cloud 版本要看兼容关系。不要随便混。

## 小结

Seata 适合解决跨服务数据库操作的一致性问题。

但它不是微服务的默认必需品。

先判断业务到底需要：

```text
强一致
还是最终一致
```

如果最终一致能接受，消息和补偿往往更简单。

如果必须强一致，再认真评估 Seata 的 AT / TCC / Saga / XA 方案。
