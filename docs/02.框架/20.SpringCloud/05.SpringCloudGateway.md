# Spring Cloud Gateway

官方文档：[Spring Cloud Gateway](https://docs.spring.io/spring-cloud-gateway/reference/)

Spring Cloud Gateway 是 Spring Cloud 体系里常用的网关。

它的作用不是“转发请求”这么简单。实际项目里，网关通常负责：

- 统一入口。
- 路由转发。
- 鉴权前置。
- 统一跨域。
- 限流。
- 请求/响应改写。
- 灰度路由。

## 三个核心概念

Gateway 里最重要的是：

```text
Route      路由规则
Predicate 断言，判断请求是否匹配
Filter    过滤器，对请求或响应做处理
```

一次请求进来，大概流程是：

```text
请求 -> Predicate 匹配 Route -> 执行 Filter -> 转发到目标服务
```

## 引入依赖

```xml
<dependency>
  <groupId>org.springframework.cloud</groupId>
  <artifactId>spring-cloud-starter-gateway-server-webflux</artifactId>
</dependency>
```

注意：Gateway 基于 WebFlux / Reactor Netty，不要和传统 Spring MVC 网关思路混在一起。

## 最小路由配置

```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: user-route
          uri: http://localhost:8081
          predicates:
            - Path=/users/**
```

意思是：请求路径匹配 `/users/**` 时，转发到 `http://localhost:8081`。

## 配合服务发现

如果接了 Nacos/Eureka，可以用服务名：

```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: user-route
          uri: lb://user-service
          predicates:
            - Path=/users/**
```

`lb://` 表示走 Spring Cloud LoadBalancer。

这时 Gateway 会根据服务名找到具体实例。

## Predicate 示例

Path：

```yaml
predicates:
  - Path=/api/users/**
```

Method：

```yaml
predicates:
  - Method=GET,POST
```

Header：

```yaml
predicates:
  - Header=X-Request-Id, \d+
```

Query：

```yaml
predicates:
  - Query=token
```

Predicate 的作用就是判断“这个请求是否命中这条路由”。

## Filter 示例

去掉路径前缀：

```yaml
filters:
  - StripPrefix=1
```

比如请求：

```text
/api/users/1
```

`StripPrefix=1` 后转发给下游的是：

```text
/users/1
```

添加请求头：

```yaml
filters:
  - AddRequestHeader=X-Gateway, spring-cloud-gateway
```

重写路径：

```yaml
filters:
  - RewritePath=/api/(?<segment>.*), /${segment}
```

## 全局过滤器

如果要做统一鉴权、日志、trace，可以写 `GlobalFilter`。

```java
@Component
public class AuthGlobalFilter implements GlobalFilter, Ordered {
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        String token = exchange.getRequest().getHeaders().getFirst("Authorization");
        if (token == null || token.isBlank()) {
            exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
            return exchange.getResponse().setComplete();
        }
        return chain.filter(exchange);
    }

    @Override
    public int getOrder() {
        return -100;
    }
}
```

这是一个很简单的示例。真实项目里不要只判断 token 是否存在，还要校验签名、过期时间、权限等。

## 常见坑

### 1. Gateway 和 Spring MVC 混用

Gateway 是 WebFlux 栈。不要在同一个应用里随便混 Spring MVC 和 Gateway，依赖冲突会很烦。

### 2. 路径转发多一段或少一段

大多数是 `Path`、`StripPrefix`、`RewritePath` 没配清楚。

排查时直接打印下游收到的 path。

### 3. 鉴权写在每个服务里

公共鉴权适合放 Gateway 做第一层。

但这不代表下游服务完全不校验权限。网关是入口防线，不是唯一防线。

## 小结

Gateway 先记住：

```text
Route      转发到哪里
Predicate 什么时候转发
Filter    转发前后做什么
```

服务名路由用 `lb://service-name`。
