# 熔断降级 Resilience4j

官方文档：[Spring Cloud CircuitBreaker](https://docs.spring.io/spring-cloud-circuitbreaker/reference/)

老项目里经常看到 Hystrix。

现在新项目更应该看 Spring Cloud CircuitBreaker + Resilience4j。

## 为什么需要熔断

微服务里，一个请求可能会调用很多服务：

```text
order-service -> user-service -> coupon-service -> payment-service
```

如果 `coupon-service` 卡住，调用方线程一直等，最后可能把 `order-service` 也拖死。

熔断的目标是：

```text
下游明显不稳定时，先快速失败或降级，避免故障扩散
```

## 引入依赖

```xml
<dependency>
  <groupId>org.springframework.cloud</groupId>
  <artifactId>spring-cloud-starter-circuitbreaker-resilience4j</artifactId>
</dependency>
```

如果是 WebFlux 项目，还可以使用 reactive 相关依赖。

## 使用 CircuitBreakerFactory

```java
@Service
public class UserService {
    private final RestTemplate restTemplate;
    private final CircuitBreakerFactory<?, ?> circuitBreakerFactory;

    public UserService(RestTemplate restTemplate,
                       CircuitBreakerFactory<?, ?> circuitBreakerFactory) {
        this.restTemplate = restTemplate;
        this.circuitBreakerFactory = circuitBreakerFactory;
    }

    public String getUser(Long id) {
        CircuitBreaker circuitBreaker = circuitBreakerFactory.create("userService");
        return circuitBreaker.run(
            () -> restTemplate.getForObject("http://user-service/users/" + id, String.class),
            throwable -> "用户服务暂时不可用"
        );
    }
}
```

fallback 里不要做复杂逻辑。越简单越好。

## 配置 Resilience4j

```yaml
resilience4j:
  circuitbreaker:
    instances:
      userService:
        slidingWindowSize: 10
        minimumNumberOfCalls: 5
        failureRateThreshold: 50
        waitDurationInOpenState: 10s
```

几个常见配置：

| 配置 | 含义 |
| --- | --- |
| `slidingWindowSize` | 滑动窗口大小 |
| `minimumNumberOfCalls` | 最少多少次调用后才计算失败率 |
| `failureRateThreshold` | 失败率超过多少触发熔断 |
| `waitDurationInOpenState` | 熔断打开后等待多久再尝试半开 |

## 熔断状态

Resilience4j 常见状态：

```text
CLOSED      正常调用
OPEN        熔断打开，直接失败或降级
HALF_OPEN   半开，允许少量请求试探恢复
```

不要把熔断理解成“服务失败就立刻永远不调用”。它是动态状态。

## 限流、重试、隔离

Resilience4j 不只熔断，还提供：

- Retry
- RateLimiter
- Bulkhead
- TimeLimiter

但是不要全部一股脑加上。

比如：

- 读接口可以考虑重试。
- 写接口慎用重试，避免重复写。
- 高并发入口可以限流。
- 慢调用可以配超时。

## 常见坑

### 1. fallback 吞掉所有问题

fallback 不能只是返回“成功”。否则上游以为业务成功，实际数据已经错了。

### 2. 写操作随便重试

下单、支付、扣库存这类操作，重试前要先考虑幂等。

### 3. 熔断阈值拍脑袋

阈值要结合业务和监控调整。没有观测数据，配置只是猜。

## 小结

熔断降级不是为了让错误消失，而是为了控制错误的影响范围。

优先记住：

```text
超时必须配
fallback 要简单
写操作重试要谨慎
熔断阈值要结合监控调
```
