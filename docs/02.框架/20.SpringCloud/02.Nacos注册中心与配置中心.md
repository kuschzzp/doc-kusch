# Nacos 注册中心与配置中心

官方文档：[Nacos Spring Cloud Quick Start](https://nacos.io/en/docs/quick-start-spring-cloud)

Nacos 在 Spring Cloud Alibaba 体系里很常见。

它主要做两件事：

```text
服务发现：服务注册、服务查询、健康检查
配置管理：集中配置、动态刷新、环境隔离
```

## 服务注册发现

引入依赖：

```xml
<dependency>
  <groupId>com.alibaba.cloud</groupId>
  <artifactId>spring-cloud-starter-alibaba-nacos-discovery</artifactId>
</dependency>
```

配置：

```yaml
server:
  port: 8081

spring:
  application:
    name: user-service
  cloud:
    nacos:
      discovery:
        server-addr: 127.0.0.1:8848
```

启动类可以加：

```java
@EnableDiscoveryClient
@SpringBootApplication
public class UserApplication {
    public static void main(String[] args) {
        SpringApplication.run(UserApplication.class, args);
    }
}
```

现在很多场景下，只要依赖和配置正确，服务发现会通过自动配置生效。

`@EnableDiscoveryClient` 更多是显式表达“这是一个服务发现客户端”。

## 服务名很重要

```yaml
spring:
  application:
    name: user-service
```

这个名字会注册到 Nacos。

后续其他服务调用时，会用这个服务名，而不是写死 IP：

```text
http://user-service/users/1
```

这背后会走服务发现和负载均衡。

## Nacos Config

引入依赖：

```xml
<dependency>
  <groupId>com.alibaba.cloud</groupId>
  <artifactId>spring-cloud-starter-alibaba-nacos-config</artifactId>
</dependency>
```

配置 Nacos 地址：

```yaml
spring:
  application:
    name: user-service
  cloud:
    nacos:
      config:
        server-addr: 127.0.0.1:8848
        file-extension: yaml
```

Nacos 配置里常见 Data ID 规则：

```text
${prefix}-${spring.profiles.active}.${file-extension}
```

默认情况下，`prefix` 通常来自 `spring.application.name`。

比如：

```text
user-service-dev.yaml
user-service-prod.yaml
```

## namespace、group、dataId

Nacos 配置里这三个概念很重要。

| 概念 | 作用 |
| --- | --- |
| namespace | 环境隔离，比如 dev、test、prod |
| group | 业务分组，默认 `DEFAULT_GROUP` |
| dataId | 一份具体配置文件的标识 |

我的习惯：

```text
namespace 按环境分
Group 不要乱用，除非确实有业务隔离需求
Data ID 按服务名 + profile 管
```

## 动态刷新

Nacos Config 支持配置动态刷新。

如果某个 Bean 需要刷新，可以使用：

```java
@RefreshScope
@RestController
public class ConfigController {
    @Value("${user.level:normal}")
    private String userLevel;

    @GetMapping("/level")
    public String level() {
        return userLevel;
    }
}
```

配置中心适合放会变的运行配置，不适合放所有东西。

比如数据库密码、密钥这类敏感信息，要结合权限、加密和密钥管理方案，不要随便明文放。

## 常见坑

### 1. 服务注册了，但调用不到

先看：

- 服务名是否一致。
- namespace 是否一致。
- group 是否一致。
- Nacos 控制台健康状态是否正常。
- 调用方是否引入 LoadBalancer 或 Feign 相关依赖。

### 2. 配置拉不到

先看：

- Data ID 是否符合规则。
- file extension 是否一致。
- namespace id 是否填对。
- group 是否一致。
- profile 是否激活。

### 3. 本地配置和远程配置谁优先

不同版本和配置方式会影响优先级。不要靠猜，项目里最好明确约定：哪些配置只允许在 Nacos 改，哪些允许本地覆盖。

## 小结

Nacos 的两个核心能力：

```text
Discovery：服务注册发现
Config：配置集中管理
```

实际排查问题时，优先看三件事：

```text
服务名
namespace
group / dataId
```
